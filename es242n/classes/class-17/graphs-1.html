Modern computing leans on graph algorithms — PageRank, GPS routing, recommendation engines, dependency resolution, network routing. Today we’re building the foundation: how to store a graph, and how to walk one. Friday: how to use those tools to decide things about graphs (shortest path, cycles, components).

Quick — fewest hops from A to H?

Eight stations. Some direct connections (the lines). How many hops from A to H?

Three hops — three shortest paths (longer paths exist too)

You just did BFS

What you did to find “fewest hops”: you fanned out from A — first the neighbours, then the neighbours of those, then the next level out — until you hit H.

That algorithm has a name: Breadth-First Search. You've already met it on grids: explore outward from a starting cell, level by level, until you reach the target. Same idea, messier picture.

Three hops because H is at level 3 from A. The three paths exist because the earlier levels have several vertices — any of them can carry you to the next level.

The levels, made explicit

BFS visits the graph in levels: every vertex one hop away, then every vertex two hops away, then three. The level of a vertex equals its shortest-path distance from the source. That's why BFS finds the fewest hops — by construction.

What’s new — and what isn’t

You’ve already used BFS, DFS, and a visited set on grid problems. So what actually changes when we generalise to graphs?

Tree

Neighbours via pointers.
Acyclic — no visited set needed.

Grid

Neighbours implicit: (i±1, j±1).
Cycles exist → visited set needed.

Graph

Neighbours must be stored.
Cycles exist → visited set needed.

The one new thing graphs add: the neighbours have to be written down. BFS, DFS, the visited bolt — same as grids. Today’s only new work: how to store the edges so we can ask neighbors(v).

The cycle trap — A, B, C, A, B, C, …

Pretend you didn’t track visited nodes. Start at A and follow edges:

A → B → C → A → B → C → A → …

The three edges of the triangle form a cycle. Any cycle, no matter how big, has this property: you can leave a node and come back to it.

Same fix as grids: mark each vertex visited the moment you enqueue it, and never enqueue a visited vertex again. Trees didn’t need this (no cycles); grids and graphs both do.

Part 1

Vocabulary, and the interface BFS / DFS will use

Vertices, edges, neighbours

  • Vertex (or node) — a circle. The graph above has 8 vertices: A, B, C, D, E, F, G, H.
  • Edge — a line connecting two vertices. The graph has 12 edges.
  • Neighbours — two vertices joined by an edge. B’s neighbours (green): A, C, E. So B has three of them.
  • Degree — how many neighbours a vertex has. degree(B) = 3.

Notation: a graph is G = (V, E), a set of vertices and a set of edges. Today |V| = 8, |E| = 12.

Path, cycle, connected

  • Path — a sequence of vertices where each consecutive pair shares an edge. A → C → F → H is a path of length 3. Length = number of edges.
  • Cycle — a path that starts and ends at the same vertex (using at least one edge, never repeating one). A → B → C → A is a cycle of length 3.
  • Connected — for every pair of vertices, there is a path between them. Our graph is connected: from any node you can reach any other.
A tree from last week, in graph language: a graph that is connected and acyclic — no cycles, every vertex reachable. Friday we’ll spot trees inside graphs by checking these two properties.

Two knobs every graph has

Directed vs undirected

If edges have a direction (one-way streets, follower-on-Twitter, web links), the graph is directed. Drawn with arrowheads.

Our graph today is undirected: every edge works both ways. If A connects to B, then B connects to A.

Weighted vs unweighted

If each edge carries a number (distance, time, cost), the graph is weighted. Otherwise every edge is “one hop.”

Today: unweighted. We’ll add weights on Friday when we get to Dijkstra.

Four combinations: undirected / unweighted (today), directed / unweighted, undirected / weighted, directed / weighted. Same vocabulary; different algorithms cope with different combinations.

The graph, as an abstract type

Same trick we played with queues and stacks: before deciding how to store a graph, decide what we need to ask of one. BFS and DFS will work on this — and nothing more:

class Graph:
    vertices()                // an iterable of all vertices
    neighbors(v)              // an iterable of v's neighbours
    has_edge(u, v)            // True if u and v are connected directly

That’s the entire interface. Three things you can ask of a graph.

BFS and DFS only ever call neighbors(v). They never look inside how the graph is stored. Write BFS against this interface today, and any implementation underneath gives the same answer — different cost, identical result.

Part 2

Use the interface — BFS and DFS, with the visited bolt

BFS — what you did, written down

When you found “A to H in 3 hops,” you had two unconscious lists going:

  • A list of nodes to explore next — first A’s neighbours (B, C, D), then theirs.
  • A list of nodes already seen, so you didn’t revisit B from C.

Make those two lists explicit and you have BFS:

To explore: a queue

First-in-first-out, so we drain a whole level before starting the next. That’s what makes BFS layered.

Already seen: a visited set

Mark a vertex visited the moment you enqueue it. Never enqueue a visited vertex again.

BFS — pseudocode

BFS(graph, source):
    visited ← empty set
    Q ← empty queue

    visited.add(source)                        // mark BEFORE enqueueing
    Q.enqueue(source)

    while Q not empty:
        u ← Q.dequeue()
        visit u                                // print, count, whatever you came for

        for each v in graph.neighbors(u):      // ← only call into the graph
            if v not in visited:
                visited.add(v)
                Q.enqueue(v)
The only line that talks to the graph is graph.neighbors(u). The rest is just a queue, a set, and a loop. BFS doesn’t know — or care — how the graph is stored.

Queue: [ ]

Why O(V + E)? Each vertex is enqueued at most once (the visited check forbids a second entry) — so the queue does at most V dequeues. For each dequeued vertex, we scan its neighbour list. The total length of all neighbour lists is 2|E| (each undirected edge contributes to two lists). So total work = V dequeues + 2E neighbour visits = O(V + E).

Drop the visited set — what happens?

Start BFS from A, but skip the if v not in visited check. Just enqueue every neighbour, every time.

Dequeue order: A, B, C, D,  C, B, E,  B, A, F,  A, G,
               A, C, E,  H, …  (and it keeps going — A and B
               and C will keep landing back in the queue forever)
The triangle does this: A enqueues B, C, D. B enqueues A again (because the edge B–A exists and we didn’t check). C enqueues A and B again. Round and round.

The visited set isn’t a polish on BFS — it’s load-bearing. Without it, a graph with even one cycle never terminates.

DFS — same algorithm, swap the container

BFS used a queue, so it drained level by level. DFS uses a stack — or, equivalently, the call stack of recursion. So instead of finishing a level before moving on, it dives all the way down one path, then backtracks.

BFS and DFS visit the same vertices. They differ in the order. BFS goes by distance from source; DFS goes by “keep following one path until you can’t.”

Both still need the visited set. The cycle trap doesn’t care which container you use.

DFS — pseudocode (recursive)

DFS(graph, source):
    visited ← empty set
    DFS-VISIT(graph, source, visited)

DFS-VISIT(graph, u, visited):
    visited.add(u)
    visit u

    for each v in graph.neighbors(u):          // ← only call into the graph
        if v not in visited:
            DFS-VISIT(graph, v, visited)

The recursion is the stack. Each recursive call pushes a frame; each return pops one. If you wrote this iteratively, you’d replace the queue in BFS with an explicit stack — and only that.

Same single point of contact with the graph: graph.neighbors(u).

Visit order: A → B → C → F → E → H → G → D

Visited: {A, B, C, F, E, H, G, D}

Same O(V + E) reasoning as BFS: each vertex is visited exactly once (the visited check forbids re-entry), and for each vertex DFS scans its neighbour list. Total work = V visits + 2E neighbour scans = O(V + E). Same complexity as BFS — just a different visit order.

Both visit every vertex once and every edge twice (undirected) — that’s the O(V + E). Both call graph.neighbors(u) and nothing else. The difference is purely in order.

Part 3

Now implement the interface — two ways

How does neighbors(v) actually work?

BFS and DFS asked the graph for neighbors(v) and never cared how the answer was produced. Now we owe an answer. Two standard ways to store a graph — both common, both worth knowing:

Adjacency matrix

A 2-D array M. M[u][v] = 1 if there’s an edge from u to v, else 0.

Adjacency list

An array of lists. adj[u] is the list of u’s neighbours.

Same interface — both can answer neighbors(v) — different storage, different costs. Let’s build both on our running graph.

Adjacency matrix

Symmetric across the diagonal because the graph is undirected. The 24 ones = 12 edges, counted from each end.

What the matrix is good (and bad) at

  • has_edge(u, v) — look at M[u][v]. O(1).
  • neighbors(u) — scan its whole row of length V. O(V) even if u has only 2 neighbours.
  • Space — O(V²). For a graph with 1 million vertices, that’s a trillion cells. Often wasteful, since most are 0.
Adjacency matrix is great when the graph is dense (most pairs are connected) and you need fast “is there an edge?” checks. For traversal — where BFS/DFS keep calling neighbors(u) — the matrix is wasteful.

Adjacency list

A → [B, C, D]
B → [A, C, E]
C → [A, B, F]
D → [A, G]
E → [B, F, H]
F → [C, E, G, H]
G → [D, F, H]
H → [E, F, G]

An array of lists. Total list entries = 2 × |E| = 24 (each edge appears in two lists, once for each endpoint).

What the list is good (and bad) at

  • neighbors(u) — return adj[u] directly. O(degree(u)). Tight — only the actual neighbours.
  • has_edge(u, v) — scan adj[u] for v. O(degree(u)), slower than the matrix.
  • Space — O(V + E). Beautiful for sparse graphs (most real graphs).
Adjacency list is great for sparse graphs and traversal — BFS/DFS enumerate neighbours of every vertex they visit, and the list answers neighbors in exactly the time the answer needs. That’s why most graph algorithms use lists.

Matrix vs list — when to pick which

The punchline: the BFS and DFS we wrote in Part 2 work without change for both storages. Swap matrix for list — BFS keeps running, just with different cost on each neighbors call. That’s what an interface buys you.

Coding exercise — the Graph interface in C

A Graph is an opaque type — you pass Graph * around without ever seeing inside. BFS and DFS call only three functions on it.

Mandatory — what BFS / DFS will call

// graph.h

typedef struct Graph Graph;        // opaque

int  graph_vertex_count(const Graph *g);
//   → returns n. BFS / DFS need it
//     to allocate visited[n].

int  graph_neighbors(const Graph *g, int v,
                     int *out, int max);
//   → fills out[0..count-1] with v's
//     neighbours; returns count.

bool graph_has_edge(const Graph *g,
                    int u, int v);
//   → true if u and v share an edge.

Helpers — design these yourself

Not part of the interface. They’re scaffolding so your code can construct, populate, and clean up. Names and signatures are your call.

  • Constructor(s) — one per backing, or one with a flag
  • add_edge / load-from-file
  • Destroy / free
  • Anything else you need (degree, print, …)
Mandatory functions are the contract BFS and DFS depend on — change them and every algorithm breaks. Helpers are scaffolding; they can vary freely.

Two backings — same interface, different storage

Backing 1 — adjacency matrix

struct Graph {
    int  n;
    int *matrix;   // flat n*n array
                   // matrix[u*n + v] = 1 or 0
};
  • has_edge(u, v) → matrix[u*n + v]. O(1).
  • neighbors(v, out, max) → scan row of length n. O(n).
  • Space: O(n²).

Backing 2 — adjacency list

struct Graph {
    int   n;
    int **list;        // list[v] = array of v's
                       //   neighbours
    int  *list_count;  // list_count[v] = deg(v)
};
  • has_edge(u, v) → scan list[u]. O(deg(u)).
  • neighbors(v, out, max) → copy list[v]. O(deg(v)).
  • Space: O(V + E).
The three mandatory functions sit on top — identical signatures, identical contract. The struct underneath changes. BFS and DFS, written against the interface, don’t notice the swap.

BFS in C — talks only to the interface

void bfs(const Graph *g, int source) {
    int   n       = graph_vertex_count(g);
    bool *visited = calloc(n, sizeof(bool));
    int  *queue   = malloc(n * sizeof(int));
    int   head    = 0, tail = 0;

    visited[source] = true;
    queue[tail++]   = source;

    int neigh[n];                                       // scratch buffer

    while (head < tail) {
        int u = queue[head++];
        printf("%d\n", u);                              // "visit" u

        int count = graph_neighbors(g, u, neigh, n);    // ← only call into the graph
        for (int i = 0; i < count; i++) {
            int v = neigh[i];
            if (!visited[v]) {
                visited[v] = true;
                queue[tail++] = v;
            }
        }
    }
    free(visited); free(queue);
}
bfs uses only graph_vertex_count and graph_neighbors. Link it with the matrix backing — it runs. Link it with the list backing — same code, same answer, different cost.

Build it — and feel sparse vs dense in your runtime

  1. Write the matrix backing — the three mandatory functions plus whatever helpers you need (constructor, add_edge, destroy).
  2. Write the list backing — same three mandatory functions, different storage.
  3. Write bfs as above. (DFS too if you finish early — recursive, same pattern.)
  4. Build the small graph from class (8 vertices, 12 edges) on both backings. Verify BFS prints the same sequence either way.
Bonus — time them: a sparse graph (V = 1000, E ≈ 2000) versus a dense one (V = 1000, E ≈ 100 000). Which backing wins on which? That’s “sparse vs dense” in your runtime, not just in the slides.

What we built today

  • The frame: graph = traversal you already know + 2 bolts (representation, visited).
  • Vocabulary: vertex, edge, neighbour, degree, path, cycle, connected; directed/undirected; weighted/unweighted.
  • The interface: vertices(), neighbors(v), has_edge(u, v). BFS and DFS use only neighbors.
  • BFS: queue + visited. Layer by layer. Finds shortest paths in unweighted graphs.
  • DFS: recursion (or stack) + visited. Same vertices, different order. Natural for cycle and component problems.
  • Implementation: adjacency matrix (dense, fast edge-query) vs adjacency list (sparse, fast neighbour-listing). Same interface, two different storages.
BFS and DFS are the only two traversal algorithms we’ll need. Everything Friday is one of these two, with one more line of bookkeeping bolted on.
