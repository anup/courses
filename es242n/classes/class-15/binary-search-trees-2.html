Part 1

Delete — the hardest BST operation

Why delete is hard

Search and insert each followed one path from root to leaf, doing one thing at the end.

Delete has to do more:

  1. Find the node to delete.
  2. Remove it from the tree.
  3. Re-attach its children (and their subtrees) so the BST property still holds.

Step 3 is where the difficulty lives. What we do depends on how many children the deleted node has.

Three cases, by child count: 0 children (leaf), 1 child, 2 children.

Case 1 — the node is a leaf

Before

Delete 20.

After

Set 30's left pointer to NULL.

The easy case. Nothing to re-attach — just set the parent's pointer to NULL.

Case 2 — the node has one child

Before

Delete 70. It has only one child, 80.

After

80 takes 70's place. (Or: 50's right child becomes 80.)

Replace the node with its only child. The child slides up into the deleted node's slot. BST property is preserved automatically.

Case 3 — naive promotion breaks

The deleted node has two children. Try the strategy from Case 2 — promote one of them — and watch what happens.

Delete 50. Children: 30 (left) and 70 (right).

Promote 30?

  • 30 slides up to become the new root.
  • But 70 and its subtree (60, 70, 80) are left floating — their parent (50) is gone.
  • 70's entire subtree is orphaned.

Promote 70?

  • 70 slides up.
  • Now 30's subtree (20, 30, 40) is orphaned.
Promotion only works when the deleted node has at most one subtree. With two, one of them is always left without a parent.

What we really need

Instead of moving a whole subtree, try a different idea: replace just the deleted node's value, not its position. The tree's shape stays the same; only the value at one slot changes. Both subtrees stay attached exactly where they were.

For the BST property to still hold, the replacement value must:

  • Already exist somewhere in the tree (so we don't conjure new values).
  • Be larger than every value in the left subtree (preserves "left < me").
  • Be smaller than every other value in the right subtree (preserves "me < right").
Only two values in the entire tree satisfy all three conditions: the deleted node's inorder successor and its predecessor.

Part 2

Inorder successor and predecessor

What is the inorder successor?

The inorder successor of a node N is the value that comes right after N in the inorder traversal.

Equivalently: the smallest value in the tree that is greater than N's value.

Inorder of this tree: 20, 30, 40, 50, 60, 70, 80

So the successor of 30 is 40 (the next value emitted).

The predecessor of 30 is 20.

Why successor works as the replacement

Recall the three conditions: the replacement must be greater than every value in the left subtree, smaller than every other value in the right subtree, and already in the tree.

The successor of the deleted node N is — by definition — the smallest value strictly greater than N. Walk the conditions:

  • Greater than N → greater than every value in N's left subtree (all of which are < N). ✓
  • Smallest value greater than N → every other value greater than N is bigger. So the successor is smaller than the rest of the right subtree. ✓
  • Already in the tree. ✓

The successor is exactly the value that fits at N's position. Predecessor works too, by symmetry.

Finding the successor — two cases

Case A — node has a right subtree

The successor is the leftmost node in the right subtree.

Example: successor of 30 is 40 — the leftmost node in 30's right subtree.

Case B — no right subtree

Walk up until you find an ancestor that you came from on the left. That ancestor is the successor.

Example: successor of 40 is 50. 40 has no right child; walking up, 40 is the right child of 30, then 30 is the left child of 50 — so 50 is the successor.

Case A is the common one and the one we use for deletion (the deleted node has two children, so it always has a right subtree). Case B requires a parent pointer or a search-from-root.

Successor — pseudocode (Case A only)

For deletion, the node we care about always has a right subtree — so we only need Case A.

SUCCESSOR-IN-RIGHT-SUBTREE(node):
    // assumes node.right ≠ NULL
    current ← node.right
    while current.left ≠ NULL:
        current ← current.left
    return current

Walk into the right subtree, then keep going left. The leftmost node there is the successor. Time: O(h).

Predecessor — mirror image

Predecessor is the value that comes right before N in inorder. Or: the largest value less than N's value.

The two cases mirror successor:

  • Node has a left subtree → predecessor is the rightmost node in the left subtree.
  • No left subtree → walk up to the first ancestor reached from the right.

In the canonical tree, predecessor of 50 is 40 — the rightmost node in 50's left subtree.

std::map gives you dhokha

If you've used C++'s std::map or Java's TreeMap:

What it looks like

  • Key → value lookup
  • Feels like a dictionary
  • Easy to assume it's a hash table

What it actually is

  • A balanced BST — red-black tree
  • O(log n) for every operation
  • Inorder iteration gives keys in sorted order

Why it matters:

  • map.begin() = smallest key. map.rbegin() = largest. (BST min and max.)
  • ++iterator = move to the next key in sorted order. (Inorder successor.)
  • map.lower_bound(x) = smallest key ≥ x. (A successor-of-x walk.)

The real hash table is std::unordered_map (C++) and HashMap (Java). Use that when you want O(1) and don't care about order.

The name "map" lies. Every ++it you've ever written on a std::map is an inorder-successor walk you didn't know you were doing.

Part 3

Delete — completing Case 3

Case 3 — using the successor

We have the tool. To delete a node N with two children:

  1. Find N's inorder successor: walk right once, then keep going left. For N = 50: leftmost of 50's right subtree is 60.
  2. Copy 60's value into the node currently holding 50.
  3. Delete the original 60 from the right subtree. 60 was a leaf — falls into Case 1.

Before

Successor of 50 is 60.

After

60 now occupies 50's slot. The original 60 leaf is gone. Both subtrees still attached.

Delete — the algorithm in two phases

To delete value v from a BST rooted at node:

Phase 1 — find the target

  • If v < node.data → descend left.
  • If v > node.data → descend right.
  • If v = node.data → we've found the target. Hand off to Phase 2.

Phase 2 — remove the target

  • Case 1 or 2 (0 or 1 child): replace the node with its (possibly NULL) only child.
  • Case 3 (2 children): copy the successor's value into this node, then recursively delete the successor from the right subtree.
Both phases are recursive. Phase 1 returns up the call chain. Phase 2's Case 3 calls back into Phase 1 to delete the successor — and that recursion always falls into Case 1 or 2, so it terminates.

Delete — pseudocode

DELETE(node, v):
    if node = NIL: return NIL

    // ─── Phase 1: walk down to find the target ───
    if v < node.data:
        node.left  ← DELETE(node.left, v)
        return node
    if v > node.data:
        node.right ← DELETE(node.right, v)
        return node

    // ─── Phase 2: v = node.data; this is the target ───
    // Case 1 (leaf) and Case 2 (one child) collapse to:
    if node.left  = NIL: return node.right
    if node.right = NIL: return node.left

    // Case 3: two children. The successor trick.
    succ ← LEFTMOST(node.right)
    node.data ← succ.data                          // copy successor's value
    node.right ← DELETE(node.right, succ.data)     // delete original successor
    return node
Why return node? Each call returns its (possibly modified) subtree to its caller. The caller's node.left ← DELETE(...) reattaches the child slot to whatever the recursion gave back.

Delete — trace on the canonical tree

Call DELETE(root, 50) on the canonical tree. Watch the recursion:

  1. At 50: v = node.data. Phase 2. Both children present → Case 3.
  2. Successor: LEFTMOST(50.right) = LEFTMOST(70) = 60.
  3. Copy: root.data ← 60. Tree now has 60 at root and 60 in the right subtree.
  4. Recurse on right subtree: DELETE(70, 60).
    • At 70: 60 < 70 → Phase 1 left. 70.left ← DELETE(60, 60).
    • At 60: v = node.data. Phase 2. No children → return NIL.
    • Back at 70: 70.left = NIL. Return 70.
  5. Back at root: root.right ← 70. Return root.

Delete — complexity

Each step costs at most O(h):

  • Finding the node — one path from root: O(h).
  • Case 1 or 2 — constant work after finding.
  • Case 3 — find successor (O(h)) + one recursive delete (O(h)). The recursion does Case 1 or 2.
Total: O(h). Same shape as search and insert.

All four BST operations — search, insert, delete, kth smallest — bottom out at the same cost: a walk down the tree.

Part 4

Validate BST

The question

Given a binary tree, is it a BST?

You are handed a tree built by someone else. You need to decide: does it satisfy the BST property at every node?

The naïve check — and why it fails

Naïve check: at every node, verify left.data < node.data < right.data.

This tree passes every parent-child comparison:

  • 5 < 10 ✓
  • 15 > 10 ✓
  • 6 < 15 ✓
  • 20 > 15 ✓

But it's not a BST — 6 sits in the right subtree of 10, and 6 < 10.

Parent–child checks miss distant violations. The property says "every descendant," not "the immediate child."

Approach 1 — inorder walk must be sorted

We proved on Wednesday: inorder of a BST is sorted. So the converse check works: if inorder is strictly increasing, the tree is a BST.

prev ← −∞
VALIDATE(node):
    if node = NULL: return TRUE
    if not VALIDATE(node.left): return FALSE
    if node.data ≤ prev: return FALSE
    prev ← node.data
    return VALIDATE(node.right)

The variable prev tracks the last value emitted by the inorder walk. If the next value isn't strictly greater, the BST property is broken somewhere.

Approach 2 — pass min/max bounds down

Each node has a range it must live in. The root is unbounded; every child inherits a tighter range from its parent.

VALIDATE(node, lo, hi):
    if node = NULL: return TRUE
    if node.data ≤ lo or node.data ≥ hi: return FALSE
    return VALIDATE(node.left,  lo,         node.data)
       and VALIDATE(node.right, node.data,  hi)

// initial call:
VALIDATE(root, −∞, +∞)

The range encodes the BST property literally: "every descendant on the left must be less than me; on the right, greater than me."

Both approaches compared

ApproachHowTimeExtra memory
Inorder + prevuses the punchline (inorder = sorted)O(n)O(h) recursion stack + one variable
Min/max boundsencodes the BST property directlyO(n)O(h) recursion stack

Both visit every node once. Both are correct. The bounds approach is more "literal"; the inorder approach is more "structural" and reuses Wednesday's lemma.

On Wednesday we proved inorder of a BST is sorted. Today we use the converse: a tree whose inorder is sorted is a BST. Same fact, used both ways.

Part 5

Tree problems

Lowest common ancestor (LCA)

Given a BST and two values u and v (both present in the tree), find their lowest common ancestor — the deepest node that has both in its subtree.

LCA of 20 and 40:

  • At root 50 — both targets are smaller. Go left.
  • At 30 — 20 is smaller, 40 is larger. Targets split here. 30 is the LCA.

LCA — three-case recurrence

At each node, compare both targets to the node's value:

  • Both targets smaller than node → LCA is in the left subtree.
  • Both targets larger than node → LCA is in the right subtree.
  • Targets split (one ≤ node ≤ other) → this node is the LCA.
LCA(node, u, v):
    if u < node.data and v < node.data:
        return LCA(node.left, u, v)
    if u > node.data and v > node.data:
        return LCA(node.right, u, v)
    return node

Time: O(h). In a general (non-BST) binary tree, LCA is O(n) — you have to compare paths. The BST property turns this into a walk.

Sorted array → balanced BST

Given a sorted array, build a balanced BST containing the same values.

Idea: pick the middle element as the root. Everything to its left in the array becomes the left subtree (built recursively); everything to its right becomes the right subtree.

BUILD-BST(arr, lo, hi):
    if lo > hi: return NULL
    mid ← (lo + hi) / 2
    node ← newNode(arr[mid])
    node.left  ← BUILD-BST(arr, lo,    mid − 1)
    node.right ← BUILD-BST(arr, mid + 1, hi)
    return node

Each level of recursion halves the input. Result: a balanced tree with height ≤ ⌈log₂ n⌉. Total time: O(n).

Sorted array → balanced BST — example

Input: [1, 2, 3, 4, 5, 6, 7]. Middle is 4 → root.

  • Left half [1, 2, 3] → middle 2 → root of left subtree.
  • Right half [5, 6, 7] → middle 6 → root of right subtree.
  • Same pattern continues until each piece is a single leaf.

Height 2 on 7 nodes — perfectly balanced. Inorder of this tree: 1, 2, 3, 4, 5, 6, 7 — the original sorted input.

Recover BST — fixing a corrupted tree

Someone hands you a BST where two values were accidentally swapped. Restore the BST without rebuilding it from scratch.

The fix: a valid BST's inorder walk is strictly increasing. Two swapped values leave one or two descending steps.

  • Swapped nodes adjacent in inorder → one descending pair. Swap those two values.
  • Swapped nodes apart → two descending pairs. Swap the first value of pair 1 with the second value of pair 2.

Inorder of this tree: 6, 2, 3, 4, 1. Descending pairs: (6, 2) and (4, 1). Swap 6 ↔ 1; inorder becomes 1, 2, 3, 4, 6.

Recover BST — pseudocode

first  ← NIL
second ← NIL
prev   ← NIL

INORDER(node):
    if node = NIL: return
    INORDER(node.left)
    if prev ≠ NIL and prev.data > node.data:
        if first = NIL: first ← prev     // first violation
        second ← node                     // most recent violation
    prev ← node
    INORDER(node.right)

RECOVER-BST(root):
    INORDER(root)
    swap(first.data, second.data)

One inorder pass with a running prev finds the two values to swap. Time: O(n). Space: O(h) recursion stack.

Wednesday's lemma — inorder of a BST is sorted — used as a diagnostic. Every place the inorder is broken is a place the BST is wrong, and inorder tells you exactly which two values to fix.

What we built today

  • Inorder successor and predecessor — the "next" and "previous" values in sorted order.
  • Delete in three cases, with successor replacement for the two-children case.
  • Two ways to validate that a binary tree is a BST.
  • LCA in O(height) using the BST property.
  • Building a balanced BST from a sorted array.
  • Recovering a corrupted BST by reading its inorder for descending steps.
All four CRUD operations are now O(height). And the inorder-equals-sorted lemma from Wednesday has earned its keep — validating BSTs, finding kth smallest, and now restoring broken ones.
