S1

Session 1 — Delete, the tool that solves it, then delete again

Open with deletion as the hardest BST operation. Walk the three cases by child count: leaf (just unlink), one child (promote the child — sliding it up one level), two children (try promoting either child and watch a whole subtree get orphaned). Hitting that wall is what motivates the next tool: a single replacement value with very specific properties — must be already in the tree, larger than the entire left subtree, smaller than every other value in the right subtree.

That motivation introduces the inorder successor and predecessor. Define what each is, then prove successor fits the deletion requirement exactly. Two cases for finding successor (with right subtree → leftmost of it; without → walk up to a left-coming ancestor). Pseudocode for the deletion-relevant case. A short detour: std::map and TreeMap are not hash tables — they're red-black BSTs, and every ++iterator on them is an inorder-successor walk you've been calling without knowing.

Then return to Case 3 with the successor in hand: copy the successor's value to the deleted node, recursively delete the original successor (which falls into Case 1 or 2). Full recursive pseudocode. Complexity: O(height), same as search and insert.

S2

Session 2 — Validate BST and tree problems

Validate-BST as the converse of Wednesday's "inorder of a BST is sorted" punchline. Open by recalling the local-vs-global trap (the same counterexample tree from Part 1) — students see why the obvious parent-child check is wrong. Two correct approaches: inorder walk with a running prev, and min/max bounds passed down the recursion. Compare both — both O(n), both correct.

Three tree problems applying everything:

  1. Lowest common ancestor — in a general binary tree this is O(n) (compare two paths). In a BST, the property turns it into a three-case walk down the tree: O(height).
  2. Sorted array → balanced BST — pick the middle as root, recurse on halves. Total O(n), height ≤ ⌈log₂ n⌉.
  3. Recover BST — restore a BST whose two values were swapped. One inorder pass spots the descending steps; the two values to swap fall out. Showcases Wednesday's inorder = sorted lemma used as a diagnostic.

Close with the take-away: all four CRUD operations (search, insert, delete, kth smallest) are O(height). The BST property does the heavy lifting in each of them.

In-class exercises

  • Find the inorder successor and predecessor of every node in the canonical tree.
  • Delete 30 from the canonical tree by hand; identify which case applies and execute it.
  • Trace the validate-BST function (both approaches) on the trap tree from Part 1.
  • Find LCA of 20 and 60 in the canonical tree.
  • Build a balanced BST from [1, 2, 3, 4, 5, 6, 7] and confirm its height is 2.
  • Given a BST with two values swapped, write its inorder, identify the descending pair(s), and pick the two values to swap back.
