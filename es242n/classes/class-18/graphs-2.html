Part 1

Connected components — how many separate pieces?

How many separate pieces?

Three connected components: {A,B,C,D}, {E,F,G}, {H,I}.

A connected component is a maximal set of vertices that are all reachable from each other.

The graph above has three components. Picking any vertex and doing DFS/BFS from there gives you one component. To get all of them: DFS in a loop over every unvisited vertex.

Real-world: social-network islands, road-network connected regions, components of a circuit, equivalence classes — anywhere “reachable from” defines a partition.

The algorithm — DFS in a loop

int count_components(const Graph *g) {
    int n = graph_vertex_count(g);
    bool *visited = calloc(n, sizeof(bool));
    int components = 0;

    for (int v = 0; v < n; v++) {
        if (!visited[v]) {
            dfs(g, v, visited);     // marks v and everything reachable from it
            components++;           //   as visited — that’s one whole component
        }
    }

    free(visited);
    return components;
}
Each DFS visit walks an entire component and marks every vertex visited. The outer loop just counts how many times we had to start fresh.

Cost: each vertex is visited exactly once across all DFS calls (visited check). Each edge is inspected twice. O(V + E) total — the loop adds no extra asymptotic cost.

Trace — three components

  1. v = A. Not visited → DFS from A. Walks A, B, C, D — marks all four visited. components = 1.
  2. v = B, C, D → all visited, skip.
  3. v = E. Not visited → DFS from E. Walks E, F, G. components = 2.
  4. v = F, G → visited, skip.
  5. v = H. Not visited → DFS from H. Walks H, I. components = 3.
  6. v = I → visited, skip. Loop ends.
Final answer: 3 components. The outer loop fired exactly three times.

Part 2

Cycle detection — does this graph have a loop?

What does DFS “see” when it finds a cycle?

Run DFS from any vertex on a graph with a cycle. At some point during the walk, you’ll try to recurse into a neighbour and find it already visited. That’s a back edge — and a back edge means you’ve found a cycle.

● Green = DFS tree edges  ·  ┄ Dashed orange = back edge (closes a cycle)

DFS from A walks A → B → D → C. From C, we try to recurse into A — already visited.

That visit is a back edge from C to A. The cycle is A → B → D → C → A.

The rule: if DFS ever finds a neighbour already in the visited set, you have a cycle. Almost. There’s a catch.

The catch — undirected edges look like cycles to themselves

You walk into B from A. From B, A is a neighbour and is visited. Is that a cycle?

No. It’s the same edge you just walked. In an undirected graph, every edge looks like a back edge if you ignore which direction you came from.

The fix is one extra parameter: carry the parent vertex into the recursive call. When you find a visited neighbour, only flag a cycle if that neighbour isn’t your parent.

The rule, corrected: a visited neighbour that isn’t your parent is a back edge — and that’s a cycle.

DFS + visited + parent — pseudocode

bool dfs_cycle(const Graph *g, int u, int parent, bool *visited) {
    visited[u] = true;

    int neigh[g->n];
    int count = graph_neighbors(g, u, neigh, g->n);

    for (int i = 0; i < count; i++) {
        int v = neigh[i];
        if (!visited[v]) {
            if (dfs_cycle(g, v, u, visited)) return true;   // cycle found downstream
        } else if (v != parent) {
            return true;                                    // ← back edge: a cycle!
        }
    }
    return false;
}

bool has_cycle(const Graph *g) {
    int n = graph_vertex_count(g);
    bool *visited = calloc(n, sizeof(bool));
    for (int v = 0; v < n; v++)
        if (!visited[v])
            if (dfs_cycle(g, v, -1, visited)) { free(visited); return true; }
    free(visited);
    return false;
}

The outer loop handles disconnected graphs — a cycle in any component is a cycle for the whole graph. Pass parent = -1 at the root because the source has no parent.

Part 3

Is this graph a tree? — the consolidation problem

What makes a graph a tree?

Three equivalent characterizations — any two of these imply the third for a graph with n vertices:

Connected

From any vertex, you can reach any other — exactly one component.

Acyclic

No back edges anywhere — DFS from any vertex never sees a visited non-parent neighbour.

Exactly n − 1 edges

Just enough to wire the tree together. One fewer and it’s disconnected; one more and there’s a cycle.

Picking your two: the cleanest pairing is connected + acyclic — both directly tested by DFS with one bool flag. The edge-count check is a faster sanity test if you only have |E| handy.

The consolidation — is_tree

bool is_tree(const Graph *g) {
    if (count_components(g) != 1) return false;   // must be one piece
    if (has_cycle(g))             return false;   // must be acyclic
    return true;
}

// or equivalently, in one walk:

bool is_tree_one_pass(const Graph *g) {
    int n = graph_vertex_count(g);
    bool *visited = calloc(n, sizeof(bool));

    // start from vertex 0; if any cycle, fail
    if (dfs_cycle(g, 0, -1, visited)) { free(visited); return false; }

    // if any vertex remained unvisited, the graph is disconnected
    for (int v = 0; v < n; v++)
        if (!visited[v]) { free(visited); return false; }

    free(visited);
    return true;
}
One DFS from any starting vertex does both jobs at once: it finds cycles (back-edge check) and tells you which vertices are reachable (visited array). If both pass — no cycle, all visited — the graph is a tree.

Part 4

Shortest path on an unweighted graph — BFS, but remember the distance

BFS levels are distances

You saw it on Wednesday: BFS visits vertices in layers, and a vertex’s layer index equals its shortest-hop distance from the source.

Wire that into the algorithm and you have a shortest-path computer:

int *bfs_distances(const Graph *g, int source) {
    int n = graph_vertex_count(g);
    int *dist = malloc(n * sizeof(int));
    for (int i = 0; i < n; i++) dist[i] = -1;   // unreachable

    int *queue = malloc(n * sizeof(int));
    int head = 0, tail = 0;

    dist[source] = 0;                             // ← only line that's new
    queue[tail++] = source;

    int neigh[n];
    while (head < tail) {
        int u = queue[head++];
        int count = graph_neighbors(g, u, neigh, n);
        for (int i = 0; i < count; i++) {
            int v = neigh[i];
            if (dist[v] == -1) {                  // ← “unvisited” check via dist
                dist[v] = dist[u] + 1;            // ← one hop further than u
                queue[tail++] = v;
            }
        }
    }
    free(queue);
    return dist;
}

The dist array doubles as the visited set: -1 means “not yet seen.”

BFS distance from A — final distances

Every vertex labelled with its shortest hop-count from A. H is 3 hops away — which we already knew from Wednesday’s opener, but now the algorithm tells us instead of us seeing it by eye.

What if edges have weights?

Now imagine each edge carries a number: a distance, a cost, a time. The “shortest path” isn’t fewest hops anymore — it’s the smallest total weight.

Shortest path from A to D?

  • A → B → D = 2 + 3 = 5
  • A → C → D = 5 + 1 = 6
  • A → B → C → D = 2 + 8 + 1 = 11
BFS picks A → B → D (5). But that’s 2 hops vs A → C → D’s 2 hops. BFS can’t distinguish — it would settle for either. Worse, on a graph where the cheap path takes more hops, BFS picks the wrong one.

Part 5

Dijkstra — BFS with a priority queue

BFS’s queue assumes “one hop = one unit”

BFS drains the queue in FIFO order. That works because every dequeue moves the “frontier” forward by exactly one hop — so the first time you dequeue a vertex, you’ve found the shortest path to it.

With weighted edges, FIFO doesn’t know that. The next vertex to finalise is the one with the smallest total distance, which might be three hops away through cheap edges or one hop away through a single expensive one.

The fix: swap the queue for a min-heap keyed on tentative distance. Always pop the vertex with the smallest distance — that’s the next one to finalise.

Same algorithm. One container swap. The min-heap from Monday is doing real work now.

Dijkstra — the steps

  1. Start with dist[source] = 0, all others = ∞.
  2. Put (0, source) into a min-heap, keyed on distance.
  3. While the heap isn’t empty:
    • Pop the vertex u with the smallest distance.
    • If u is already finalised, skip.
    • Otherwise, mark it finalised. For each neighbour v of u:
      • If dist[u] + weight(u, v) < dist[v], update dist[v] and push (dist[v], v) onto the heap.
The relaxation step — if (dist[u] + w < dist[v]) dist[v] = dist[u] + w — is the heart of Dijkstra. Each relaxation tightens our upper bound on the shortest distance to v. Finalising = no more relaxation is possible.

Dijkstra — pseudocode

int *dijkstra(const WGraph *g, int source) {
    int n = wgraph_vertex_count(g);
    int *dist = malloc(n * sizeof(int));
    bool *done = calloc(n, sizeof(bool));
    for (int i = 0; i < n; i++) dist[i] = INT_MAX;

    MinHeap *pq = heap_create();
    dist[source] = 0;
    heap_push(pq, 0, source);                 // (priority, vertex)

    while (!heap_empty(pq)) {
        int u = heap_pop(pq);                 // smallest tentative distance
        if (done[u]) continue;
        done[u] = true;

        WEdge neigh[n];
        int count = wgraph_neighbors(g, u, neigh, n);
        for (int i = 0; i < count; i++) {
            int v = neigh[i].to;
            int w = neigh[i].weight;
            if (dist[u] + w < dist[v]) {       // ← relaxation
                dist[v] = dist[u] + w;
                heap_push(pq, dist[v], v);
            }
        }
    }
    heap_destroy(pq);  free(done);
    return dist;
}

A vertex may be pushed onto the heap more than once (each relaxation queues a new entry). The done check on pop discards stale entries — the first time we pop a vertex, its distance is final.

Dijkstra — trace

  1. Pop A (dist 0). Relax B → 2, C → 5. Heap: [(2,B), (5,C)].
  2. Pop B (dist 2). Relax C? 2+8=10, worse — skip. Relax D → 5. Heap: [(5,C), (5,D)].
  3. Pop C (dist 5). Relax D? 5+1=6, worse than 5 — skip. Heap: [(5,D)].
  4. Pop D (dist 5). No new improvements. Heap empty. Done.
Final: dist[A]=0, dist[B]=2, dist[C]=5, dist[D]=5. Note: A→B→D and A→C→D both reach D at total 5 — Dijkstra picks the first one it finalises, and both are correct.

Why O((V + E) log V)?

  • Each vertex is finalised at most once (the done check). So at most V pops.
  • Each edge can trigger at most one relaxation — and each relaxation pushes once onto the heap. So at most E pushes.
  • Every heap push and pop costs O(log V).
  • Total: (V + E) × O(log V) = O((V + E) log V).
This is where Monday’s heap pays off. Without a heap, you’d scan the whole “unfinalised” list each time looking for the minimum — that’s O(V²). For sparse graphs (most real-world ones), the heap version wins by a lot.

Why the unit ends here: Dijkstra is BFS + heap + relaxation. Three things you already own, composed into the most-used shortest-path algorithm in the world. GPS, routing protocols, anything with weighted reachability — Dijkstra under the hood.
