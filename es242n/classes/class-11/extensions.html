Two extensions, one template

Both today's permutations and what follows here use the choose-recurse-undo skeleton. What changes between problems is:

  • What counts as a choice.
  • What counts as a complete answer.

Everything else — the state, the recursion, the undo — survives intact.

Subsets

Same template, different choices

The problem

Given {A, B, C, D}, print every subset.

{ }
{ A }     { B }     { C }     { D }
{ A B }   { A C }   { A D }   { B C }   { B D }   { C D }
{ A B C } { A B D } { A C D } { B C D }
{ A B C D }

How many? 16 subsets — including the empty set and the full set.

In general: 2n subsets.

Why 2n?

For each element, we make one decision: include it, or don't. Two choices per element.

n elements × 2 choices each = 2n distinct decision sequences.

Each unique sequence of decisions defines one subset.

Permutations: n choices for the first slot, then n-1, then n-2... → n!.
Subsets: 2 choices per element, every element gets a choice → 2n.

The shape of the search

Binary choice at every level. Depth n. Leaves are subsets.

For {A, B, C}: 2³ = 8 leaves. Each path encodes one yes/no decision per element.

What changes from permutations?

PermutationsSubsets
Choices per slotn (any unused letter)2 (include or skip)
Need used[]?Yes — to avoid reusing lettersNo — each element is visited exactly once, in order
What we buildA length-n permutationA variable-length subset
Output countn!2n

The template — choose, recurse, undo — survives intact. Only the meaning of "choose" changes.

The code

int arr[N], chosen[N];
int size = 0;                       // size of the current subset

void subsets(int depth, int n) {
    if (depth == n) {
        printf("{ ");
        for (int i = 0; i < size; i++) printf("%d ", chosen[i]);
        printf("}\n");
        return;
    }
    // Choice 1: skip arr[depth]
    subsets(depth + 1, n);

    // Choice 2: include arr[depth]
    chosen[size++] = arr[depth];
    subsets(depth + 1, n);
    size--;                         // undo
}

Same shape as permute(). Two recursive calls (the two choices) instead of a loop. size-- is the undo.

Why no used[]?

In permutations, every recursive call can pick any unused letter — choices and positions are decoupled.

In subsets, we descend element by element in order. Element depth is the only one we're deciding about at depth depth. The next call moves to depth + 1. We never revisit an element.

The state for "have I used this element?" is implicit in depth itself. We don't need a separate flag.

Permutations II

What happens when the input has duplicates

The setup

Run our permute() on input {1, 1, 2}. What does it print?

1 1 2
1 2 1
1 1 2     ← duplicate of line 1
1 2 1     ← duplicate of line 2
2 1 1
2 1 1     ← duplicate of line 5

Six lines. Three of them duplicates.

Why? Because the algorithm treats arr[0] = 1 and arr[1] = 1 as different positions — even though they hold equal values.

The unique answer is 3 permutations

1 1 2
1 2 1
2 1 1

We want to print each distinct sequence exactly once.

Naive de-duplication (store all outputs, sort, dedup) would work but is wasteful. We want to prune the search tree: skip the branches that would produce duplicates before we generate them.

The fix — two parts

Part 1. Sort the input. Equal values are now adjacent.

arr = [ 1, 1, 2 ]    // already sorted in this example

Part 2. Add one line inside the loop, right after the existing used[i] check:

if (i > 0 && arr[i] == arr[i-1] && !used[i-1]) continue;

That's it. The whole fix.

Why does that line work?

After sorting, equal elements sit next to each other. Consider the moment we're filling slot depth, looping over i, and we reach two equal elements at arr[i-1] and arr[i].

If used[i-1] = 1

Then arr[i-1] is currently on the path, at some earlier slot. Putting arr[i] at this slot is a different permutation. Keep it.

If used[i-1] = 0

Then we already tried arr[i-1] at this slot and backtracked. Trying arr[i] (equal value) at the same slot would produce the same subtree of outputs. Skip.

Equal values at the same slot are interchangeable. We only let the first equal value at any slot proceed.

Trace on {1, 1, 2}

At depth 0, we loop i = 0, 1, 2:

  • i = 0: take arr[0] = 1. Recurse. (Prints "1 1 2" and "1 2 1".) Undo.
  • i = 1: arr[1] == arr[0] and used[0] = 0 (we just undid it). Skip.
  • i = 2: take arr[2] = 2. Recurse. (Prints "2 1 1".)

Three permutations. No duplicates.

One line changed. The tree gets exactly the branches we want and skips the rest.
