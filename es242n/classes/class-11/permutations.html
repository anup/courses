Part 2

The one insight

If you knew how to list permutations of two letters,
could you list permutations of three?

Look at the three-letter list again

A | B C        ← A first, then perms of {B, C}
A | C B
B | A C        ← B first, then perms of {A, C}
B | C A
C | A B        ← C first, then perms of {A, B}
C | B A

The list splits into three groups of two.

Read the bar again

Every line is:

(a first letter)  |  (a permutation of the rest)

Permutations of three letters = pick one letter, then list permutations of the other two.

That's a recursive recipe

To permute {A, B, C}:

  1. Pick A first. List permutations of {B, C}. Glue A in front of each.
  2. Pick B first. List permutations of {A, C}. Glue B in front.
  3. Pick C first. List permutations of {A, B}. Glue C in front.

Each smaller problem is exactly the same kind of problem — just with one fewer letter.

And what about {B, C}?

The same trick again:

  1. Pick B first. List permutations of {C}. Glue B in front.
  2. Pick C first. List permutations of {B}. Glue C in front.

And {C} is one letter — just C. Done.

Every recursion needs a stopping point. Ours is: "one letter — just print it."

Part 4

The tree the recursion walks

Every choice opens a branch

Six green leaves. Six permutations. One per path.

Where are the boxes?

At every node, PLACED is whatever's written inside that node.

AVAILABLE is "the original letters" minus "what's in the node."

The recursion never stores the tree. The tree is the picture of what the recursion does. Each node is a function call.

Part 5

From paper to code

PLACED in code

PLACED is an ordered list. In C, we use an array.

Call it chosen[].

chosen[0] = B
chosen[1] = A
chosen[2] = C

The length of the placed list = how deep we are. We'll call that depth.

AVAILABLE in code

One way: keep a real list, remove from it, put it back.

Easier way: keep a flag for each letter — "still available" or "taken."

used[0] = 0   ← A is available
used[1] = 1   ← B is taken
used[2] = 0   ← C is available

The available letters are the ones with used[i] == 0.

Why used[] instead of a real list?

If we removed B from a list, we'd have to slide elements around. And then put B back.

Flipping a flag is one operation. No sliding.

We don't actually remove anything. We just mark what's no longer available. The recursion sees the array the same way it always did.

Three pieces of state

On paperIn codeWhat it tells us
PLACEDchosen[]What's in the prefix, in order
AVAILABLEused[]Which letters are no longer pickable
PositiondepthWhich slot we're filling right now

Three things. That's the whole state.

Part 6

What does one step do?

Pretend you are permute(depth)

You have been called. Your job:

  1. Look at slot depth.
  2. Try every available letter in that slot.
  3. For each try, ask the recursive call to handle the rest.
  4. Then put the letter back so the next try is clean.

That's it.

In plain English

for each letter i:
    if i is taken: skip
    place i at slot depth
    mark i as taken
    ask the recursion to fill slots after depth
    un-mark i

Six lines. The five words "ask the recursion to fill" are doing the heavy lifting.

One thing to notice

The loop is the only place we make a choice. The recursion never picks for us; it just fills the rest.

Every node in the tree is one entry to permute(). The for-loop inside it makes the branches.

Part 7

Trust the call

When we write permute(depth + 1)...

...what are we trusting that call to do?

Print every permutation that starts with the values already in chosen[0..depth].

We do not trace inside the call. We trust it.

Why this is the hard part

If you try to imagine the recursion fully expanded, you'll see hundreds of calls and panic.

Instead, ask:

  • "What does this one call do?"
  • "What does it promise the smaller call will do?"
Define one level. Trust the level below. The whole tree falls out for free.

Part 8

Why "undo" matters

After the recursive call returns

The recursion has printed everything that starts with the letters we placed.

Now we want to try a different letter in slot depth.

But our last choice is still in chosen[depth], and still marked in used[].

If we don't put things back, the next iteration of the loop sees the wrong state.

What "undo" does

One line. Right after the recursive call:

used[i] = 0;

That's it. We're un-marking the letter we tried so the next loop iteration starts fresh.

We don't bother clearing chosen[depth] — the next iteration will overwrite it anyway.

What if we forget?

Try this thought experiment.

Pick A → recurse → it prints A B C and A C B → returns.

Loop wants to try B. But used[A] = 1 still — we never un-marked A.

The recursion at depth 1 also sees A as taken, even though it's been removed from slot 0 already. The whole subtree is wrong.

In practice: the program prints far fewer permutations than it should, or crashes.

Choose.
Recurse.
Undo.

Then try the next choice.

Part 9

The code, line by line

Step 1: the signature

void permute(int depth, int n) {

}

depth = which slot we're filling. n = total number of slots.

Step 2: the base case

void permute(int depth, int n) {
    if (depth == n) {
        for (int i = 0; i < n; i++) printf("%d ", chosen[i]);
        printf("\n");
        return;
    }
}

When all slots are filled, print and stop.

Step 3: the loop over choices

void permute(int depth, int n) {
    if (depth == n) { /* print and return */ }

    for (int i = 0; i < n; i++) {
        if (used[i]) continue;     // skip taken letters

    }
}

Try every letter arr[i]. Skip the ones already in use.

Step 4: choose

    for (int i = 0; i < n; i++) {
        if (used[i]) continue;
        chosen[depth] = arr[i];    // place it in this slot
        used[i] = 1;               // mark it taken

    }

The "choose" step is two lines: place into chosen, flip the flag in used.

Step 5: recurse

    for (int i = 0; i < n; i++) {
        if (used[i]) continue;
        chosen[depth] = arr[i];
        used[i] = 1;
        permute(depth + 1, n);     // trust this call

    }

One line. We trust it to fill the rest.

Step 6: undo

    for (int i = 0; i < n; i++) {
        if (used[i]) continue;
        chosen[depth] = arr[i];
        used[i] = 1;
        permute(depth + 1, n);
        used[i] = 0;               // ← undo
    }

One line. Un-mark the letter so the next iteration is clean.

The whole thing

int arr[N], chosen[N], used[N];

void permute(int depth, int n) {
    if (depth == n) {
        for (int i = 0; i < n; i++) printf("%d ", chosen[i]);
        printf("\n");
        return;
    }
    for (int i = 0; i < n; i++) {
        if (used[i]) continue;
        chosen[depth] = arr[i];
        used[i] = 1;
        permute(depth + 1, n);
        used[i] = 0;
    }
}

Fifteen lines. The seven inside the for-loop are the whole pattern.

Part 11

When it goes wrong

The four common bugs

What you seeThe cause
Only one permutation printsForgot used[i] = 0
Repeated letters in outputMissing if (used[i]) continue
Stack overflow / crashMissing or wrong base case
All on one lineForgot printf("\n")
Three of these four are bugs in state management. The fourth is just printing. Backtracking is mostly state bookkeeping.

The diagnostic move

If your output is wrong, add one print at the top of permute():

void permute(int depth, int n) {
    for (int i = 0; i < depth; i++) printf("%d ", chosen[i]);
    printf("\n");
    /* ... rest as before ... */
}

Now you see every node of the tree, in the order the recursion visits them. Bugs jump out.

Typing reference

#include <stdio.h>
#define N 20

int arr[N], chosen[N], used[N];

void permute(int depth, int n) {
    if (depth == n) {
        for (int i = 0; i < n; i++) printf("%d ", chosen[i]);
        printf("\n");
        return;
    }
    for (int i = 0; i < n; i++) {
        if (used[i]) continue;
        chosen[depth] = arr[i];
        used[i] = 1;
        permute(depth + 1, n);
        used[i] = 0;
    }
}

int main(void) {
    int n;
    scanf("%d", &n);
    for (int i = 0; i < n; i++) scanf("%d", &arr[i]);
    permute(0, n);
    return 0;
}
