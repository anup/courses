Permutations

Building the intuition

ES 242(N) · Class 11 · 8 June 2026

The challenge

Print all permutations of {1, 2, 3, 4}.

The iterative approach needs four nested loops, with skip-if-equal conditions everywhere:

for (i = 0; i < n; i++)
  for (j = 0; j < n; j++) { if (j == i) continue;
    for (k = 0; k < n; k++) { if (k == i || k == j) continue;
      for (l = 0; l < n; l++) { if (l == i || l == j || l == k) continue;
        printf("%d %d %d %d\n", a[i], a[j], a[k], a[l]);
}}}

For arbitrary n, you cannot write n nested loops — n is a runtime value.

Fixed-depth iteration. Variable-depth problem. We need a different tool.

Why n!?

One way to see it: build big from small.

Suppose we already have all six permutations of {A, B, C}:

A B C    B A C    C A B
A C B    B C A    C B A

To get permutations of {A, B, C, D}, take each one and insert D in every position:

A B C  →  D A B C   A D B C   A B D C   A B C D     (4 new)
A C B  →  D A C B   A D C B   A C D B   A C B D     (4 new)
…

4 positions × 6 permutations = 24. Add E: 5 × 24 = 120. That's why permutations(n) = n × permutations(n−1) = n!.

That's recursive — but going the other way is easier to code

The "insert into every gap" view builds all permutations of n-1 first, stores them, then expands. To print 120 perms of 5 items, we'd hold all 24 perms of 4 in memory first.

Same recursion, opposite direction: instead of adding the last letter, decide the first letter, then recurse on what's left.

Same recurrence, n × T(n−1). One direction stores the small ones. The other prints one permutation at a time.

Look inside one group

Take the six that start with A:

A B C D     A C B D     A D B C
A B D C     A C D B     A D C B

Strip the A:

B C D       C B D       D B C
B D C       C D B       D C B

Those are exactly the six permutations of {B, C, D}.

Permutations starting with A = A, then every permutation of the remaining letters.

The same problem, smaller input

permutations({A, B, C, D}) =
    A · permutations({B, C, D})  ∪
    B · permutations({A, C, D})  ∪
    C · permutations({A, B, D})  ∪
    D · permutations({A, B, C})

Each line is the same problem with one fewer letter.

That's recursion. The function calls itself, on a smaller input, with confidence that the call will work.

What does the recursive call promise?

"permute({remaining letters}) prints every permutation of those letters."

That's the contract. The whole job of one call to permute:

  • Pick a first letter from what's left.
  • Hand the remaining letters to the recursive call.
  • Trust it to print every permutation of those.
  • Move on to the next first letter.
Define what one call promises. Trust the recursion to keep that promise. We never trace inside the call.

Concretely — pick D first

Fix D in slot 0. Call permute({A, B, C}). By the promise, it prints:

A B C        B A C        C A B
A C B        B C A        C B A

With D in front of each:

D A B C      D B A C      D C A B
D A C B      D B C A      D C B A

Six permutations starting with D. Repeat with A, B, C as the first letter. 24 total.

We wrote exactly one level of the algorithm — pick a letter, call, repeat. The recursion writes the rest.

The recursion's invisible drawing

Each call is a node. Each "pick a letter" is a branch. Each leaf is a permutation.

Six leaves for three letters. n! leaves for n.

Watch the recursion walk it

Each click: descend to a leaf, print, then back up and try the next branch. The blue path is "where the recursion is right now."

Stand inside one call

Pretend you are permute() called somewhere in the middle of the recursion. The letters are {A, B, C, D}.

You don't know where you came from. You just have to do your job: fill the next empty slot.

What do you need to be told from outside?

  • Which slot is the next empty one.
  • What letters are already placed (so you don't reuse them).

Two pieces of information. In code, we'll keep them in three arrays — chosen[], used[], and a depth counter — because we want fast lookups, not because there's a third concept.

The path so far — chosen[]

An array of length n. We write letters into it as we place them.

If the recursion has placed B in slot 0 and A in slot 1, then:

chosen = [ 'B', 'A', _, _ ]

When all slots are filled, chosen holds one complete permutation. We print it.

chosen[] is the answer we're building. It grows as the recursion descends and shrinks (logically) as it backs up.

What's still available — used[]

An array of length n with a flag per letter.

used = [ 1, 1, 0, 0 ]
         A  B  C  D       ← A and B are taken; C and D are free

Why not just look at chosen[] to know what's used? We could — but scanning it every iteration is O(depth). used[i] is O(1).

chosen[] tracks the answer. used[] tracks the constraint. Same information from two angles — kept separately so each is fast to check.

Which slot — depth

The slot we're filling right now. It's also the number of letters already placed. Also the depth in the choice tree.

Unlike chosen and used, depth is a function parameter — each call gets its own value.

void permute(int depth, int n);   // depth comes in as a parameter

permute(0, 4)  →  permute(1, 4)  →  permute(2, 4)  →  permute(3, 4)  →  print
Every recursive call increases depth by one. When depth == n, all slots are filled.

Where do these arrays live?

Global. One copy in memory, shared across every recursive call.

int arr[N], chosen[N], used[N];   // declared once, outside any function

void permute(int depth, int n) {
    // every call to permute() reads and writes the SAME arr, chosen, used
    ...
}

We don't allocate new arrays per call. We don't pass them as parameters. They live above the recursion and every call touches the same memory.

(You could pass them as pointers instead. Same effect. Globals keep the signature clean and the cohort focused on the algorithm.)

Shared state is the whole trick

When permute(0) writes chosen[0] = 'A' and sets used[A] = 1, then calls permute(1)...

...permute(1) doesn't need to be told any of that. It just looks at chosen[] and used[] directly — the same memory permute(0) just wrote into.

That's how state flows down the recursion. Each call mutates the shared arrays; the deeper call reads the mutations as facts.

But — when the deeper call returns and the outer loop wants to try the next letter, the mutations are still there. Stale.

That's why we undo.

Choose.
Recurse.
Undo.

Then try the next choice.

Pseudocode

permute(depth):
    if depth == n:
        print chosen
        return
    for each i in 0..n-1:
        if used[i]: continue
        chosen[depth] = arr[i]; used[i] = 1   // choose
        permute(depth + 1)                     // recurse
        used[i] = 0                            // undo

The for loop is "try each first letter." The recursive call is "trust it to fill the rest." The undo is what lets the loop actually move to the next iteration.

Why undo?

Walk through what happens without it.

  1. The loop picks A for slot 0. We mark used[A] = 1 and call permute(1).
  2. permute(1) prints all permutations starting with A. Returns.
  3. Back in the loop. Now we want to try B in slot 0.
  4. But used[A] is still 1 from step 1 — the shared arrays still carry that mutation.
  5. The next call to permute(1) reads used[A] = 1 and skips A — even though slot 0 is now B, so A is actually free again.
Shared state from one iteration leaks into the next. The subtree comes out wrong.

One line — used[i] = 0 right after the recursive call — restores the state, so each iteration of the loop starts clean.

The code

#include <stdio.h>
#define N 20

int arr[N], chosen[N], used[N];

void permute(int depth, int n) {
    if (depth == n) {
        for (int i = 0; i < n; i++) printf("%d ", chosen[i]);
        printf("\n");
        return;
    }
    for (int i = 0; i < n; i++) {
        if (used[i]) continue;
        chosen[depth] = arr[i];
        used[i] = 1;
        permute(depth + 1, n);
        used[i] = 0;
    }
}

int main(void) {
    int n; scanf("%d", &n);
    for (int i = 0; i < n; i++) scanf("%d", &arr[i]);
    permute(0, n);
    return 0;
}

Four bugs and one diagnostic

What you seeCause
Only one permutationForgot used[i] = 0
Repeated lettersMissing if (used[i]) continue
Crash / overflowMissing base case
All on one lineForgot printf("\n")
Diagnostic: add printf for chosen[0..depth-1] at the top of permute(). You now see every node of the choice tree in visit order — bugs become obvious.
