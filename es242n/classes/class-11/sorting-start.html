Sorting — Part 1

Merge Sort

ES 242(N) · Class 11 (overflow) · 8 June 2026

The problem

Given an unsorted array, produce a sorted one.

input:    [ 5, 2, 8, 1, 9, 3, 7, 4 ]
output:   [ 1, 2, 3, 4, 5, 7, 8, 9 ]

You already know one or two ways to do this: bubble sort, selection sort, insertion sort. They all cost O(n²).

Today we look at a different approach that costs O(n log n) — the same recursive idea you just used for permutations, with a different shape.

Divide and conquer

If we had two sorted halves of the array, could we make the whole thing sorted?

left half (sorted):   [ 2, 5, 8 ]
right half (sorted):  [ 1, 3, 9 ]
                                   →   [ 1, 2, 3, 5, 8, 9 ]

Yes — and the procedure is straightforward. Walk both halves with two pointers; pick the smaller front each time.

Sorting n elements becomes: sort the left half, sort the right half, merge them. Each smaller problem is the same kind of problem.

The recursion writes itself

Same shape as permutations: solve n using two solutions of n/2.

mergeSort(arr, lo, hi):
    if lo >= hi: return                  // base case
    mid = (lo + hi) / 2
    mergeSort(arr, lo, mid)               // sort left
    mergeSort(arr, mid+1, hi)             // sort right
    merge(arr, lo, mid, hi)               // combine

Three lines of real work. Three things to understand: the base case, the two recursive calls, and what merge does.

How the splits look

For input [5, 2, 8, 1, 9, 3, 7, 4]:

                [5, 2, 8, 1, 9, 3, 7, 4]
                  /                     \
            [5, 2, 8, 1]             [9, 3, 7, 4]
             /        \               /         \
         [5, 2]     [8, 1]        [9, 3]      [7, 4]
          /  \       /  \          /  \        /  \
        [5] [2]    [8] [1]       [9] [3]     [7] [4]      ← base cases (length 1)

Splitting goes down. The actual sorting work happens on the way back up, in the merge step.

The merge step

This is the only new mechanic. Everything else is recursion you already know.

Merge — two sorted halves into one

Two pointers, one for each half. Take the smaller front. Repeat.

258 139 → ______

Compare 2 and 1. 1 is smaller. Take it.

258 .39 → 1_____

Now compare 2 and 3. 2 is smaller. Take it. Continue until both halves are empty.

When one half empties, copy the rest of the other half. Done.

Merge — the code

void merge(int arr[], int lo, int mid, int hi) {
    int tmp[hi - lo + 1];
    int i = lo, j = mid + 1, k = 0;

    while (i <= mid && j <= hi) {
        if (arr[i] <= arr[j]) tmp[k++] = arr[i++];
        else                  tmp[k++] = arr[j++];
    }
    while (i <= mid) tmp[k++] = arr[i++];   // leftovers from left
    while (j <= hi)  tmp[k++] = arr[j++];   // leftovers from right

    for (int x = 0; x <= hi - lo; x++)
        arr[lo + x] = tmp[x];
}

Three loops. The first picks from whichever side has the smaller front. The next two flush whatever is left of one side after the other empties. The last copies the merged result back into arr.

Merge sort — the full code

#include <stdio.h>

void merge(int arr[], int lo, int mid, int hi) {
    int tmp[hi - lo + 1];
    int i = lo, j = mid + 1, k = 0;
    while (i <= mid && j <= hi)
        tmp[k++] = (arr[i] <= arr[j]) ? arr[i++] : arr[j++];
    while (i <= mid) tmp[k++] = arr[i++];
    while (j <= hi)  tmp[k++] = arr[j++];
    for (int x = 0; x <= hi - lo; x++) arr[lo + x] = tmp[x];
}

void mergeSort(int arr[], int lo, int hi) {
    if (lo >= hi) return;
    int mid = (lo + hi) / 2;
    mergeSort(arr, lo, mid);
    mergeSort(arr, mid + 1, hi);
    merge(arr, lo, mid, hi);
}

int main(void) {
    int arr[] = {5, 2, 8, 1, 9, 3, 7, 4};
    int n = sizeof(arr) / sizeof(arr[0]);
    mergeSort(arr, 0, n - 1);
    for (int i = 0; i < n; i++) printf("%d ", arr[i]);
    return 0;
}

Trust the recursive call (again)

When we write mergeSort(arr, lo, mid), what do we trust?

"That call leaves arr[lo..mid] sorted."

That's the contract. We don't trace inside it.

Once we trust it, the rest is mechanical:

  • Sort the left half (trust).
  • Sort the right half (trust).
  • Merge — the only step we have to actually code.
Same mindset as permute(). Define what one call promises. Trust the recursion to keep that promise.

Why it's O(n log n)

At each level of the recursion, we do one merge over n total elements — O(n) work per level.

The recursion has log₂(n) levels (we halve until size 1).

Total: n × log₂(n) = O(n log n).

For n = 10⁶: bubble sort would take hours. Merge sort takes a fraction of a second.
