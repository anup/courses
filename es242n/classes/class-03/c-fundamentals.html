Five differences in one slide. Everything else this term builds on this.

Compiler vs Interpreter

Python — interpreted

Source code is read line by line, every time you run.

file.py  →  python file.py
            (reads + runs)

No separate "build" step.

C — compiled

Source code becomes a binary once. You run the binary.

file.c   →  gcc file.c
            (produces "a.out")

./a.out
            (run as many times as you want)

Compile catches errors before you run. The cost is an extra step. The benefit is speed.

Your first C program

#include <stdio.h>

int main(void) {
    printf("Hello, ES242N!\n");
    return 0;
}
  • #include <stdio.h> — brings in printf, scanf.
  • int main(void) — every C program starts here.
  • printf(...) — prints. \n is a newline (not automatic in C).
  • return 0; — tells the OS the program ended successfully.

Compile. Run.

$ gcc hello.c
$ ./a.out
Hello, ES242N!
Two steps: gcc compiles your source .c into an executable called a.out. Then you run the executable. Edit source → recompile → re-run. You'll do this 1000 times this term.

printf — format specifiers

Python: print(x) figures it out. C: you tell printf the type.

SpecifierFor typeExample
%dintprintf("%d\n", 42);
%ldlongprintf("%ld\n", n);
%ffloat, doubleprintf("%.2f\n", 3.14);
%ccharprintf("%c\n", 'A');
%sstringprintf("%s\n", "hi");

Wrong specifier → wrong output. printf("%d\n", 3.14) prints garbage.

scanf — reading input

int n;
scanf("%d", &n);

double price;
scanf("%lf", &price);
The & means "the address of". scanf needs to write into your variable, so you give it where the variable lives. We'll explain & properly when we do pointers.

For double input, use %lf (not %f). Input and output specifiers don't always match.

Why this matters

Python

n = 10**100
print(n)
# works fine

Integers grow as big as you want.

C

int n = 10000000000;
/* compile warning
   or silent overflow */

Fixed size. Going past it = undefined behavior.

INT_MAX + 1 on most CPUs wraps to INT_MIN. Your "billion plus billion" becomes negative. One of the top bugs in new C code.

Reading constraints with type sizes in mind

ALTERNATING-PARITY:  −10⁹ ≤ a[i] ≤ 10⁹
  → fits in int  (int max ≈ 2.1 × 10⁹).  Use int.

If a problem said:    −10¹² ≤ a[i] ≤ 10¹²
  → doesn't fit in int.  Use long long.
The accumulator trap: even if each value fits in int, the sum might not. N = 200,000 values each up to 10⁹ → sum can reach 2 × 10¹⁴ → needs long long.

Skill of the term: read constraints, pick the right type. Every problem.

The C for-loop has three parts

for (int i = 0; i < n; i++) {
    printf("%d\n", i);
}

Initialization

Runs once at start

int i = 0

Condition

Checked before each iteration

i < n

Update

Runs after each iteration

i++

Python's for x in collection doesn't exist in C. C's for is closer to Python's while — you manage the loop state explicitly.

C also has while — exactly like Python's

int n = 12345;
while (n > 0) {
    printf("%d\n", n % 10);
    n = n / 10;
}
Use for when you know how many times to loop (compact: init + condition + update on one line).
Use while when the stopping condition depends on the data (e.g. "until n reaches zero").

The integer-division gotcha

Python 3

7 / 2     # 3.5
7 // 2    # 3

C

7 / 2     /* 3   */
7.0 / 2   /* 3.5 */
In C, / between two ints is integer division. The fractional part is discarded. To get a real result, at least one operand must be float or double.

Want a fractional result? Make at least one operand a double: 7 / 2.0 gives 3.5.

Common bugs in your first 10 C programs

  • scanf("%d", x) — forgot the &. Compiles, crashes at runtime.
  • if (x = 5) — assignment, not comparison. Always true. Should be ==.
  • for (i = 0; i <= n; i++) — off-by-one. Goes one past the end.
  • for (i = 0; i < n; i++); — semicolon ends the loop early.
  • printf("%d", 3.14) — wrong specifier. Output is garbage.
  • Missing \n in printf — output runs together.
  • Forgot to #include <stdio.h> — implicit-declaration warning.

Use gcc -Wall. It catches half of these before you run.
