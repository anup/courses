Part 1

The BST property

The BST property

A binary tree is a Binary Search Tree when, for every node N:

  • Every value in N's left subtree is less than N's value.
  • Every value in N's right subtree is greater than N's value.
The word every matters. The property must hold not just for immediate children — for every descendant.

Assume all values are distinct. Conventions for equal keys vary.

The trap — local check is not enough

Every parent–child pair is in order:

  • 5 < 10 ✓
  • 15 > 10 ✓
  • 6 < 15 ✓
  • 20 > 15 ✓

But 6 sits in the right subtree of 10, and 6 < 10. The BST property fails.

Checking just parent–child doesn't catch this. The rule is every descendant, not "the immediate child."

Real data is dynamic

A lot of the data we actually deal with is being added to all the time:

  • Instagram — new accounts to follow, new DMs every minute; you search for a friend by username.
  • ChatGPT chats — every prompt starts a new chat; you scroll the sidebar to find that one explanation from last week.
  • WhatsApp — new conversations float to the top; you scroll or search for an old one.
  • Spotify liked songs — you keep liking new tracks; later you hunt for one specific song.

In every one of these, four operations matter:

  • search — find an item.
  • insert — add a new item.
  • delete — remove an item.
  • update — change an existing item.
We want all four to be fast. A data structure that nails one and tanks the others isn't enough.

What everything we've learned costs

insert(v) places v wherever the structure allows. Unsorted containers don't care (O(1) at end / head / top). Sorted and BST must place v at the position its value dictates. delete(v) and update(v) have to find v first — the search cost is baked in. That's why linked-list insert is O(1) but linked-list delete is O(n) — they do different work.

Stacks and queues delete in O(1) only at their one allowed end. Anywhere else is O(n). They're designed for restricted access, not general search.

Every row above the BST has at least one O(n) cell. No existing structure we know gives all four operations cheaply.

A BST is the first structure where all four CRUD operations can be sub-linear — all O(height). The only difference from Monday's binary tree is the ordering rule. That rule is what unlocks the entire row.

Part 2

Searching

The three-case recurrence

At every node, compare the target to the node's value. Three outcomes:

  • target = node.data → found.
  • target < node.data → answer (if any) lies in the left subtree.
  • target > node.data → answer (if any) lies in the right subtree.

If we hit NULL before finding the target, it isn't in the tree.

Each step throws away an entire subtree — exactly the behaviour of binary search on a sorted array.

Search — recursive

search(node, target):
    if node = NULL:           return FALSE
    if target = node.data:    return TRUE
    if target < node.data:
        return search(node.left,  target)
    else:
        return search(node.right, target)

Three lines of logic. The NULL base case handles "not present" and the bottom of any path.

Search — trace

Search 40 (present):

  • at 50: 40 < 50 → left
  • at 30: 40 > 30 → right
  • at 40: found ✓

Search 25 (absent):

  • at 50: 25 < 50 → left
  • at 30: 25 < 30 → left
  • at 20: 25 > 20 → right
  • NULL: not found ✗

Search — iterative variant

search(root, target):
    node ← root
    while node ≠ NULL:
        if target = node.data: return TRUE
        if target < node.data:
            node ← node.left
        else:
            node ← node.right
    return FALSE

Same logic, same path. No call stack — just one pointer walking down. Use either; the recursive version reads closer to the recurrence.

Search — complexity

Each step descends one level. Search touches at most one node per level.

Time = O(height).
  • Balanced tree (every level roughly full): height ≈ log₂ n → search is O(log n).
  • Degenerate tree (a chain): height = n − 1 → search is O(n).

The BST property keeps the recurrence valid. The shape of the tree decides how good it actually is.

Part 3

Inserting

Where does a new value go?

To insert v:

  1. Walk the tree exactly as if searching for v.
  2. The search reaches a NULL child somewhere (because v isn't already in the tree).
  3. Plant the new node there.
Every insertion becomes a new leaf. We never replace a node, never split a node, never re-arrange existing nodes.

Insert — recursive

insert(node, value):
    if node = NULL:
        return newNode(value)
    if value < node.data:
        node.left  ← insert(node.left,  value)
    else:
        node.right ← insert(node.right, value)
    return node

The "return the subtree" pattern: each call returns its (possibly modified) subtree to its caller, which assigns it back into left or right.

First call from outside: root ← insert(root, value). The same line handles both "tree was empty" and "tree had a root."

Build a tree from a sequence

Insertion order: 50, 30, 70, 20, 40, 60, 80

  1. 50 — tree was empty; 50 becomes the root.
  2. 30 — 30 < 50 → left of 50.
  3. 70 — 70 > 50 → right of 50.
  4. 20 — 20 < 50 → 20 < 30 → left of 30.
  5. 40 — 40 < 50 → 40 > 30 → right of 30.
  6. 60 — 60 > 50 → 60 < 70 → left of 70.
  7. 80 — 80 > 50 → 80 > 70 → right of 70.

Insert — iterative variant

insert(root, value):
    if root = NULL: return newNode(value)
    node ← root
    while TRUE:
        if value < node.data:
            if node.left = NULL:
                node.left ← newNode(value); return root
            node ← node.left
        else:
            if node.right = NULL:
                node.right ← newNode(value); return root
            node ← node.right

One pointer walking. Compared to the recursive version it has more cases to spell out — we need to check the child before stepping in, otherwise we lose the place to attach the new leaf.

Insert — complexity

Insert(v) does two things:

  1. Walk — search the tree for v. The walk ends at a NULL child (because v wasn't there).
  2. Plant — attach a new leaf at that NULL slot.

Step 1 is a search: O(height). Step 2 is one pointer assignment: O(1).

Total: O(height). Insert inherits search's complexity because insert is a search-walk plus a constant-time plant.
  • Balanced tree → O(log n) per insert.
  • Degenerate tree → O(n) per insert.

Part 4

Min, max, and the punchline

Minimum — walk all the way left

By the BST property, anything in the left subtree is smaller than the current node. So the smallest value sits at the leftmost node — the one whose left pointer is NULL.

Walk left until you can't. Done.

Maximum — walk all the way right

Symmetric. The largest value sits at the rightmost node.

Walk right until you can't.

Min / max — code

Recursive

min(node):
    if node.left = NULL:
        return node.data
    return min(node.left)

Iterative

min(node):
    while node.left ≠ NULL:
        node ← node.left
    return node.data

Both walk the leftmost path. Time = O(height). Max is the same with left replaced by right.

What does an inorder traversal of a BST print?

Inorder of a BST is sorted

Inorder traversal visits left subtree, then root, then right subtree.

Claim: on a BST, inorder visits nodes in ascending sorted order.

This is not extra work. The traversal is the same skeleton we wrote on Monday. The BST property is what makes the output come out sorted.

Why it works

Two things are true at every node N.

Inorder visits in this order:

  1. everything in the left subtree,
  2. the root N,
  3. everything in the right subtree.

BST rule says:

  • left subtree = all values < N,
  • right subtree = all values > N.

So inorder, at this level, produces: smaller stuff → N → bigger stuff. Sorted around N.

Same rule holds inside every subtree. So the "smaller stuff" piece comes out sorted, and so does the "bigger stuff" piece. Stitched together with N in the middle, the whole walk is sorted.

Demo on the tree we built

Inorder walk:

  1. left of 50 → subtree at 30
    • left of 30 → subtree at 20 → 20
    • 30
    • right of 30 → subtree at 40 → 40
  2. 50
  3. right of 50 → subtree at 70
    • 60
    • 70
    • 80

Output: 20, 30, 40, 50, 60, 70, 80

kth smallest, almost for free

Want the 5th smallest value in a tree with a million nodes? A scan visits a million. The BST visits about 25 — if it's roughly balanced.

Why so few? Inorder visits the smallest first, then the next, then the next. Walk inorder, count as you emit, stop the moment count hits k.

Why inorder hits the smallest first: it recurses LEFT before doing anything else — all the way to the leftmost node. That's the minimum. From there, the walk works its way back up and across, emitting values in increasing order.

count  ← 0
answer ← NIL

INORDER(node):
    if node = NIL or answer ≠ NIL: return       // stop after the kth
    INORDER(node.left)
    count ← count + 1
    if count = k:
        answer ← node.data
        return
    INORDER(node.right)

KTH-SMALLEST(root, k):
    INORDER(root)
    return answer

Total: O(h + k) nodes visited. For h ≈ 20 (balanced million-node tree) and k = 5, that's 25 visits.

The surprising part: the BST never sorted anything. It grew one insert at a time, each insert O(h). No "sort everything" step ever ran. The sorted order is implicit in the tree's shape — available on demand, as many values as you need, even while new ones keep arriving.

Sorted array gives O(1) kth lookup but every new value costs O(n) to insert. BST gives O(h + k) kth and O(h) insert. For data that keeps arriving, only the BST keeps up.

Part 5

Practice

Practice 1 — range query

Given a BST and two values lo, hi with lo ≤ hi, print every value v in the tree with lo ≤ v ≤ hi, in sorted order.

The straightforward way: inorder traversal, print only values in range. Always visits all n nodes.

The sharper way: use the BST property to skip subtrees that cannot contain in-range values.

Try the sharper version on the tree below. 10 minutes.

Tree to use: same {20, 30, 40, 50, 60, 70, 80}. Range: [25, 65].

Practice 1 — solution

At every node, ask three questions:

  1. Could the left subtree contain in-range values? Only if node.data > lo — otherwise everything to the left is even smaller, all out of range.
  2. Is the current node itself in range? Yes if lo ≤ node.data ≤ hi.
  3. Could the right subtree contain in-range values? Only if node.data < hi — otherwise everything to the right is even larger.

The code is exactly those three questions, in order:

rangePrint(node, lo, hi):
    if node = NULL: return
    if node.data > lo:   rangePrint(node.left,  lo, hi)     // Q1
    if lo ≤ node.data and node.data ≤ hi:
        output node.data                                      // Q2
    if node.data < hi:   rangePrint(node.right, lo, hi)     // Q3
Q1 and Q3 are where the savings live — they prune entire subtrees that can't possibly contain in-range values. Without them, this would be plain inorder + filter, visiting all n nodes.

For range [25, 65] on the canonical tree, the output is 30, 40, 50, 60 — already in sorted order, because each recursive call respects the inorder pattern (left, self, right).

Part 6

When BSTs go bad

Sorted insertion → degenerate chain

Insert 10, 20, 30, 40, 50 in that order. Each new value is larger than every previous value, so every insertion goes right.

The tree degenerates into a linked list. Height = n − 1.

Search on this tree is back to O(n). All the advantage we built up is gone.

A BST keeps the property after any insertion order — but not the shape.

Same values, two shapes

Balanced — height 2

search ≤ 3 comparisons

Degenerate — height 6

search up to 7 comparisons

Same seven values {10, 20, 30, 40, 50, 60, 70}, two completely different cost profiles.

Balanced BSTs exist (we don't cover them)

To guarantee O(log n) performance no matter the insertion order, the tree can be self-balancing: every insert and delete does a little extra work to keep the height around log n.

  • AVL trees
  • Red-black trees
  • B-trees (databases)

Not part of this course. Library containers — std::map (C++), TreeMap (Java), sortedcontainers.SortedDict (Python) — use these underneath.

Take-away: a plain BST gets you the right pattern. The worst case stays open until you balance.
