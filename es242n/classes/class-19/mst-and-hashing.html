Minimum Spanning Tree

Connect every vertex — for the smallest total weight.

The question — find the MST

The rules

  • Pick exactly n − 1 edges — here, 5.
  • Every vertex stays reachable.
  • No cycles.
  • Smallest total weight.

Real use: connect everything cheapest.

Which 5 edges minimize the total?

The MST

● A–B(2), A–D(6), B–E(3), C–F(4), E–F(1). Total = 16.

  • Cheapest edges first — if they don't make a cycle.
  • B–C (5) skipped: B and C already linked via B–E–F–C.
  • A–E (8) skipped: same reason — cycle through B.
  • D–E (7) skipped: D already linked via D–A–B–E.
  • Two systematic ways to pick those 5 edges — next.

Kruskal — sort and pick

  1. Sort all edges by weight, ascending.
  2. Walk the sorted list:
    • If adding the edge would form a cycle → skip.
    • Otherwise → add it to the MST.
  3. Stop after n − 1 edges added.
  • The cycle check is what union–find exists for — we'll build it after we see Prim.
  • Cost:
    • Sort edges: O(E log E).
    • E cycle checks, each almost O(1) with Union–Find.
    • Total: O(E log E), dominated by the sort.

Kruskal trace

Sorted edges:
E–F(1), A–B(2), B–E(3), C–F(4), B–C(5), A–D(6), D–E(7), A–E(8)

  1. E–F (1). Add.
  2. A–B (2). Add.
  3. B–E (3). Add — joins {A,B} with {E,F}.
  4. C–F (4). Add — C joins via F.
  5. B–C (5). Skip — cycle.
  6. A–D (6). Add — D joins.
  7. 5 edges = n − 1. Done. Total = 16.

Prim — grow from a seed

The algorithm

  1. Pick any starting vertex → visited set.
  2. Repeat n − 1 times:
    • Find the cheapest edge with one end in visited, one outside.
    • Add that edge. Mark the new vertex visited.

How — min-heap of candidate edges

  • Heap stores (weight, far end).
  • Visit u: push every edge from u.
  • Pop smallest. If far end visited → stale, skip. Else → add to MST.
  • Cost: O((V + E) log V). The Monday heap, third use.

Kruskal asks: does this edge form a cycle?

  • For every sorted edge (u, v) — would adding it close a cycle?
  • Naive: DFS from u through the partial tree. If you reach v, it's a cycle. O(V) per edge.
  • Over E edges → O(V·E). Too slow.
  • Better idea: don't search. Track which group each vertex belongs to.
  • Same group ⇒ cycle. Different groups ⇒ safe, and merge the two groups.
  • The data structure that does this: Union–Find.

The data structure

  • One integer array — parent[i].
  • Top row: the index i (which element).
  • Bottom row: parent[i] (who i points to).
  • A root points to itself: parent[i] == i.
  • Same root ⇒ same group.

The two operations — what they do

find(x) — "which group?"

  • Start at x. Walk up the parent chain.
  • Stop when you hit a node that points to itself — the root.
  • Return the root.
  • The root is the group's identity: same root ⇒ same group.

union(x, y) — "merge two groups"

  • Find x's root rx, and y's root ry.
  • Same root? Already in the same group — nothing to do.
  • Otherwise: point one root at the other (parent[rx] = ry). Two groups become one.

How this helps Kruskal

  • For each edge (u, v) in sorted order:
    • If find(u) == find(v) → same group already → adding this edge would form a cycle. Skip.
    • Else → add the edge to the MST and call union(u, v).

find and union — the code

The algorithm we just saw, in C. Walk-up for find; link roots for union.

int parent[N];

void init() {
    for (int i = 0; i < N; i++) parent[i] = i;   // everyone is their own root
}

int find(int x) {
    while (parent[x] != x) x = parent[x];        // walk up to the root
    return x;
}

void uf_union(int x, int y) {
    int rx = find(x);
    int ry = find(y);
    if (rx != ry) parent[rx] = ry;               // link one root under the other
}

bool same_group(int x, int y) {
    return find(x) == find(y);
}

Naive find can get slow

  • After many unions, the tree can grow tall.
  • Worst case: a long chain.
  • find walks the chain — O(n) hops.
  • Per Kruskal edge, O(n) just for the cycle check. Too slow.
  • We need to keep the tree shallow.

Path compression — flatten as you go

int find(int x) {
    if (parent[x] != x)
        parent[x] = find(parent[x]);
                       // ↑ rewire on return
    return parent[x];
}
  • Walk up to the root (normal find).
  • On the way back, point every visited node directly at the root.
  • Future finds on those nodes: 1 hop.
  • Average cost over a long sequence of operations: almost O(1).
  • This is what makes Kruskal's cycle-check fast.
  • That's MST + Union–Find done. Next: a new data-structure family.

Sets and Maps

"Is x here?" and "what's at key k?" — what we want before deciding how to build it.

Set — what we want

Operations

  • insert(s, x) — add an element
  • contains(s, x) — is x here?
  • remove(s, x) — drop x
  • size(s)

What it gives you

  • No duplicates
  • Order doesn't matter
  • Membership test is the whole point
  • Where you've seen it already: the visited set in BFS / DFS.
  • Real uses: dedupe a stream, "have I seen this email before?", spell-check dictionary.

Map — what we want

Operations

  • put(m, k, v) — store v under key k
  • get(m, k) — look up by key
  • remove(m, k)
  • contains_key(m, k)

What it gives you

  • Each key maps to exactly one value
  • Keys are unique; values can repeat
  • "Indexed lookup, but the index is any type"
  • Real uses: frequency counts, caching, "username → user record", config.
  • Same shape as Python dict, Java HashMap, C++ unordered_map.

Hashing

The implementation that gets us O(1) Set and Map. Use the key as an index — almost.

An array is the simplest Map

  • If keys are integers in [0, m), you don't need anything new.
  • The array is the Map. arr[key] stores the value. O(1) put, O(1) get.
  • This trick has a name: direct addressing. The "hash function" is just identity, h(k) = k.
  • The catch: array size = size of the key space.
  • Keys are 32-bit ints? Array of 4 billion. Strings? Impossible.
  • Hashing generalizes this — squeeze any key into a small index range.

The fix — squeeze the key

  • Pick a fixed table size m.
  • Define h(key) → [0, m).
  • Look up at arr[h(key)].
  • Any universe of keys, fixed table size.

What makes a good hash function

Deterministic

  • Same key → same index, always.
  • No randomness once you've picked the function.

Uniform

  • Spreads keys evenly across the table.
  • No "popular" slots.

Fast

  • O(1) per call.
  • If hashing is slow, you've lost the whole point.

Two simple hash functions

// integers — just modulo
int hash_int(int key, int m) {
    return (unsigned)key % m;
}

// strings — polynomial rolling hash
int hash_str(const char *s, int m) {
    unsigned h = 0;
    while (*s) {
        h = h * 31 + (unsigned)*s;     // 31 is a small odd prime
        s++;
    }
    return h % m;
}
  • Why m matters: real keys often come in patterns — multiples of 2, of 10, page sizes, byte boundaries.
  • If m shares a factor with the pattern, those keys clump on a small subset of slots and the rest stay empty.
  • Example: m = 10, keys = 10, 20, 30, … → all hash to slot 0. 9 of 10 slots dead.
  • Fix: pick m as a prime. With m = 11, the same keys cycle through all 11 slots evenly.
  • The 31 in the string hash is the same idea — a small prime that breaks up patterns.

Collisions

Two keys, same slot. Now what?

Chaining — each slot is a linked list (this is what we'll code)

  • Slot holds the head of a chain.
  • Collision? Head-insert. (Same pattern as week 1.)
  • Lookup: hash → walk the chain.

Cost per operation

  • Hash the key: O(1).
  • Walk the chain: average length = n / m — the load factor α.
  • (n keys spread across m slots ⇒ each slot has n/m keys on average.)
  • Total: O(1 + α). Keep α small ⇒ effectively O(1).

Open addressing — everyone lives in the array (alternative strategy — not coded today)

  • No chains. Every key lives directly in the array.
  • Slot taken? Probe the next slot, then the next.
  • Linear probing: try h, h+1, h+2, …
  • Lookup: probe in the same order until you find the key or hit an empty slot.

Open addressing — why deletion needs a trick

Setup: h(cat)=1, h(dog)=h(fish)=2. Inserted in order cat, dog, fish — so fish was displaced to slot 3.

1. After inserts

All three keys in. fish at slot 3 because slot 2 was taken.

2. Delete dog — without tombstone

Probe slot 2 → empty → "not found" — wrong, fish is at slot 3.

3. Delete dog — with tombstone

Probe slot 2 → tombstone → keep going → find fish at slot 3 ✓.

  • Tombstone = "was occupied, then deleted." Third slot state alongside empty / occupied.
  • Lookup walks past tombstones (treat as occupied). Insert reuses them (treat as empty).

Code it

All the pieces in hand — hash function, chains, collisions. Now build it: hash set → hash map → Two Sum.

Hash set — the struct

#define M 1031                  // prime; pick whatever size you want

typedef struct Node {
    int key;
    struct Node *next;
} Node;

typedef struct {
    Node *table[M];             // M head pointers, one chain per slot
    int size;
} HashSet;

void set_init(HashSet *s) {
    for (int i = 0; i < M; i++) s->table[i] = NULL;
    s->size = 0;
}

int hash_key(int key) { return (unsigned)key % M; }

Hash set — insert and contains

bool set_contains(HashSet *s, int key) {
    int h = hash_key(key);
    for (Node *p = s->table[h]; p != NULL; p = p->next)
        if (p->key == key) return true;
    return false;
}

void set_insert(HashSet *s, int key) {
    if (set_contains(s, key)) return;        // sets have no duplicates
    int h = hash_key(key);
    Node *node = malloc(sizeof(Node));
    node->key = key;
    node->next = s->table[h];                // head-insert (week 1 again)
    s->table[h] = node;
    s->size++;
}

Hash set — remove

bool set_remove(HashSet *s, int key) {
    int h = hash_key(key);
    Node **link = &s->table[h];              // pointer to the link to update
    while (*link != NULL) {
        if ((*link)->key == key) {
            Node *gone = *link;
            *link = gone->next;              // unlink
            free(gone);
            s->size--;
            return true;
        }
        link = &(*link)->next;
    }
    return false;
}

Hash map — add a value field

typedef struct MapNode {
    int key;
    int value;                  // ← only delta from the set
    struct MapNode *next;
} MapNode;

void map_put(HashMap *m, int key, int value) {
    int h = (unsigned)key % M;
    for (MapNode *p = m->table[h]; p != NULL; p = p->next)
        if (p->key == key) { p->value = value; return; }   // update
    MapNode *node = malloc(sizeof(MapNode));
    node->key = key;  node->value = value;
    node->next = m->table[h];
    m->table[h] = node;
}

bool map_get(HashMap *m, int key, int *out) {
    int h = (unsigned)key % M;
    for (MapNode *p = m->table[h]; p != NULL; p = p->next)
        if (p->key == key) { *out = p->value; return true; }
    return false;
}

Apply it

Two problems — one set, one map.

Contains Duplicate

Problem: given an array of integers, return true if any value appears more than once. Else false.

Walkthrough — nums = [1, 2, 3, 1]

inums[i]Set beforeAction
01{}not in set. Add 1.
12{1}not in set. Add 2.
23{1, 2}not in set. Add 3.
31{1, 2, 3}1 IS in set! Return true.

The key idea: as we walk, the set answers "have I seen this before?" in O(1).

  • Brute force: for each i, check all other j — O(n²) comparisons.
  • Hash set: one pass, n elements, each O(1) check → O(n).
bool contains_duplicate(int *nums, int n) {
    HashSet s;
    set_init(&s);
    for (int i = 0; i < n; i++) {
        if (set_contains(&s, nums[i]))
            return true;
        set_insert(&s, nums[i]);
    }
    return false;
}

Two Sum

Problem: given nums[] and target, find indices i, j such that nums[i] + nums[j] = target. (Exactly one solution exists.)

Walkthrough — nums = [2, 7, 11, 15], target = 9

inums[i]need = target − nums[i]Map beforeAction
027{}7 not in map. Add 2 → 0.
172{2 → 0}2 IS in map! Return [0, 1].

The key idea: for each nums[i], the map answers "have I already seen its partner?" in O(1).

  • Brute force: for each i, check all other j — n × n = O(n²).
  • Hash map: n iterations, each O(1) lookup → O(n). The map replaces the inner loop.
int *two_sum(int *nums, int n, int target) {
    HashMap m;
    map_init(&m);          // key → index
    for (int i = 0; i < n; i++) {
        int need = target - nums[i];
        int j;
        if (map_get(&m, need, &j)) {
            int *r = malloc(2 * sizeof(int));
            r[0] = j;  r[1] = i;
            return r;
        }
        map_put(&m, nums[i], i);
    }
    return NULL;
}

Bonus — meet count sort (we never covered it)

What count sort does

  • Sorts integers in a known range [0, k).
  • Make a count[] array of size k, all zeros.
  • Scan the input: for each value x, do count[x]++.
  • Walk count[] left to right: write count[i] copies of i to the output.
  • Cost: O(n + k).

Cost and limits

  • Scan input to fill count[]: O(n).
  • Walk count[] to emit output: O(k).
  • Total: O(n + k). No comparisons!

When it breaks

  • If k is huge (sorting 32-bit ints ⇒ 4-billion-cell count[]), this is unusable.
  • Fix: process the value one digit at a time. Each digit only has range 0–9. That's radix sort — next slide.

Radix sort — count sort, one digit at a time (optional — skip if short on time)

Sort by the least significant digit first, then the next, then the next. Each pass is a count sort on a single digit (range 0–9).

Input:170, 45, 75, 90, 802, 2, 24, 66
Pass 1 — sort by 1s digit:170, 90, 802, 2, 24, 45, 66, 75
Pass 2 — sort by 10s digit:802, 02, 24, 45, 66, 170, 75, 90
Pass 3 — sort by 100s digit:2, 24, 45, 66, 75, 90, 170, 802
Sorted:2, 24, 45, 66, 75, 90, 170, 802 ✓
  • Each pass uses count sort on ONE digit — that's why k = 10, not k = max value.
  • Each pass must be stable (keep earlier ordering for equal digits) so prior passes' work survives.
  • Cost: O(d · (n + k)) where d = number of digits, k = 10 for decimal. For practical inputs, effectively O(n).
