Before we begin

On Monday I motivated recursion by saying iteration cannot enumerate permutations.

That was wrong, and I want to correct it before we go further. Iteration can enumerate permutations.

Here's how — and this clean correction is also the setup for today's class.

An iterative way to list permutations

def permutations_iterative(s):
    result = [""]                       # start with one empty seed
    for char in s:
        next_result = []
        for partial in result:          # every partial we have
            for i in range(len(partial) + 1):
                # insert char into every gap
                next_result.append(partial[:i] + char + partial[i:])
        result = next_result
    return result

permutations_iterative("abc")
# → ['cba', 'bca', 'bac', 'cab', 'acb', 'abc']

Three loops. Fixed depth. No recursion anywhere. Works for any n.

This is exactly the "build big from small" intuition we used to explain why permutations of n = n!. Coded as a real algorithm, not just an intuition.

So what was my "n nested loops" slide about?

That slide showed one specific iterative approach — "loop over choices for slot 1, inside that loop over choices for slot 2..." — which would need n nested loops for n positions. That one doesn't generalize to arbitrary n.

But it's not the only iterative approach. The bottom-up insertion above is genuinely iterative and works for any n.

So why are we going to spend today on recursion anyway?

The real distinction

Permutations and subsets are constructive problems: every smaller solution is reusable as a building block for larger ones. Bottom-up iteration works.

Constrained backtracking — generate parens, N-Queens, word search, combination sum — is different. At each step you need to ask: "given my partial state, is going forward even worth it?"

For those, you have to track running state and decide to prune. That's what recursion's call stack does for free.

Today we'll see both sides of this line. Subsets — iteration works. Generate parens — iteration breaks. The template that handles both: choose-recurse-undo.

Subsets

A constructive problem — iteration works fine here

The problem

Given {A, B, C, D}, print every subset.

{ }
{ A }     { B }     { C }     { D }
{ A, B }  { A, C }  { A, D }  { B, C }  { B, D }  { C, D }
{ A, B, C }   { A, B, D }   { A, C, D }   { B, C, D }
{ A, B, C, D }

How many? 2n = 16.

For each element, one decision: include or skip. n decisions × 2 options = 2n outcomes.

The iterative way — build big from small

Same bottom-up idea as the permutations correction. If we know all subsets of {A, B}, we can build all subsets of {A, B, C} by:

  • keeping all the old subsets (those that don't include C),
  • and adding a copy of each with C appended (those that do include C).
// pseudo-iterative
result = [ {} ]
for each x in input:
    result = result ∪ { S ∪ {x} : S in result }

Two loops, fixed depth. Works for any n.

In C, the analog is a doubling loop with a 2-D array of subsets. Doable but bulky — the recursion is cleaner.

The recursive way — choose-recurse-undo

At each element, we have two choices: include or skip.

int arr[N], chosen[N];
int size = 0;                       // size of the current subset

void subsets(int depth, int n) {
    if (depth == n) {
        printf("{ ");
        for (int i = 0; i < size; i++) printf("%d ", chosen[i]);
        printf("}\n");
        return;
    }
    // skip arr[depth]
    subsets(depth + 1, n);

    // include arr[depth]
    chosen[size++] = arr[depth];
    subsets(depth + 1, n);
    size--;                         // undo
}

Same shape as permute(). Two recursive calls (the two choices) instead of a loop. size-- is the undo.

The choice tree for {A, B, C}

Binary tree, depth 3, 2³ = 8 leaves. Each path encodes a yes/no decision per element.

What changes from permutations?

PermutationsSubsets
Choices per slotn (any unused letter)2 (include or skip)
Need used[]?Yes — to avoid reusing lettersNo — each element is visited exactly once, in order
What we buildA length-n permutationA variable-length subset
Output countn!2n
Iteration possible?Yes (insert-into-gaps)Yes (double the list each step)

The recursive template — choose, recurse, undo — survives intact. Only "what's a choice" changes.

Why prefer the recursive version anyway?

  • Same skeleton as permute() — students reuse the mental model.
  • Memory: outputs one subset at a time and prints; doesn't store all 2n of them.
  • And — more importantly — the same skeleton handles constrained problems where iteration breaks. That's coming up next.
For subsets, both iteration and recursion work. We chose recursion because of where it goes next. That's the honest reason.

Generate Parentheses

A constrained problem — watch iteration break

The problem (LC 22)

Print all well-formed strings of n pairs of parentheses.

n = 3:    ((()))    (()())    (())()    ()(())    ()()()

5 strings. In general: the Catalan number Cn.

Every string has 2n characters, exactly n opens and n closes. Well-formed means: at every prefix, close count ≤ open count.

Try the bottom-up iterative trick

Could we build big from small here too? "I know all valid strings of length 2; extend each by every possible next character"?

For n = 2, the valid strings of length 2 are:

"(("    "()"

Extend each by '(' and ')':

"(("    →    "((("    "(()"
"()"    →    "()("    "())"      ← INVALID! prefix has more ) than (

We get an invalid string. Bottom-up extension doesn't know to stop.

Could we just filter at the end?

Yes — generate all 4n length-2n strings (every position is '(' or ')'), then filter for validity.

For n = 10: 410 = 1,048,576 strings generated, C10 = 16,796 valid. We did 60× more work than we needed.

For n = 15: 415 ≈ 109 generated, 9,694,845 valid. Wasted 100×.

Bottom-up doesn't know about constraints until after a string is fully built. The constraints would prune most of the tree if we asked them during construction.

The constraint is stateful

At any prefix, whether we can write '(' or ')' depends on what we've already written.

  • We can write '(' only if open count < n.
  • We can write ')' only if close count < open count.

These counts are running state. We need them at every step, and we need to update them every step.

Where do we keep running state during a tree walk? On the call stack.

The recursive solution

char chosen[64];

void gen(int open, int close, int n) {
    if (open + close == 2 * n) {
        chosen[2*n] = '\0';
        printf("%s\n", chosen);
        return;
    }
    if (open < n) {
        chosen[open + close] = '(';
        gen(open + 1, close, n);
    }
    if (close < open) {
        chosen[open + close] = ')';
        gen(open, close + 1, n);
    }
}

int main(void) { gen(0, 0, 3); return 0; }

Two recursive calls, each gated by a constraint. The constraint checks happen before we recurse — we never generate an invalid prefix.

The state — three things

NameWhat it holds
chosen[]The string so far.
openHow many '(' we've placed.
closeHow many ')' we've placed.

Depth is implicit: open + close = next slot to fill.

No used[] — characters come from an unlimited supply. Only the counters constrain us.

The counters live on the call stack — each recursive call receives its own open and close. That's how constraint state propagates without needing globals or explicit stacks.

Trace for n = 2

                    gen(0, 0)        ""
                   /          \
       open<n: '('              close<open: NO
              ↓
          gen(1, 0)             "("
         /          \
    open<n: '('              close<open: ')'
          ↓                            ↓
      gen(2, 0)  "(("           gen(1, 1)  "()"
      /     \                  /          \
   no '('   ')'           no '('          ')'
           ↓                 ↓                ↓
       gen(2, 1) "(()"   gen(2, 1) "()("   close fails
            ↓                  ↓
        gen(2, 2)            gen(2, 2)
        PRINT "(())"        PRINT "()()"

Two valid leaves: (()) and ()(). No invalid prefixes generated. Constraints prune as we go.

The line

When iteration works and when it doesn't

Two kinds of enumeration problem

Constructive

Every smaller solution is reusable as a building block. Build size n from size n-1 by extension.

Iteration works. Recursion also works — same skeleton, cleaner memory.

Examples: permutations, subsets, combinations (with care).

Constrained backtracking

Going forward depends on running state. Some partial answers must be abandoned mid-construction.

Iteration breaks — at best, generate-and-filter, which wastes exponentially.

Examples: generate parens, N-Queens, word search, combination sum, Sudoku.

Recursion handles both. That's why we invest in the template.

Choose-Recurse-Undo, recap

choose-recurse-undo(state):
    if complete(state): print and return

    for each legal choice in this state:        // ← constraint check here
        apply choice                            // choose
        choose-recurse-undo(extended state)     // recurse (trust it)
        unapply choice                          // undo

For permutations, "legal" = "not yet used."
For subsets, "legal" = "include or exclude" (always legal).
For generate parens, "legal" = "the constraint allows this character here."
For N-Queens, "legal" = "no queen attacks this square."

The template stays. The "legal" check changes.
