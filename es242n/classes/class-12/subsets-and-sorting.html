Subsets

Two choices per element

The problem

Given {A, B, C}, print every subset.

{ }      { A }      { B }      { C }
{ A, B } { A, C }   { B, C }   { A, B, C }

Eight subsets — 2³. For each element, one decision: in or out. Three decisions, two options each.

Recall Monday's permute()

int arr[N], chosen[N], used[N];

void permute(int depth, int n) {
    if (depth == n) {
        for (int i = 0; i < n; i++) printf("%d ", chosen[i]);
        printf("\n");
        return;
    }
    for (int i = 0; i < n; i++) {
        if (used[i]) continue;
        chosen[depth] = arr[i];
        used[i] = 1;
        permute(depth + 1, n);
        used[i] = 0;
    }
}

The template is right there: choose, recurse, undo. Three pieces of state — chosen[], used[], depth. A loop over n choices.

Subsets — the delta from permute()

Same skeleton. Three changes.

permute()

if (depth == n) {
    /* print chosen[0..n-1] */
    return;
}
for (int i = 0; i < n; i++) {
    if (used[i]) continue;
    chosen[depth] = arr[i];
    used[i] = 1;
    permute(depth + 1, n);
    used[i] = 0;
}

subsets()

if (depth == n) {
    /* print chosen[0..size-1] */
    return;
}
/* no loop — two explicit calls */

/* choice 1: skip arr[depth] */
subsets(depth + 1, n);

/* choice 2: include arr[depth] */
chosen[size++] = arr[depth];
subsets(depth + 1, n);
size--;
  1. Loop disappears. Two specific choices per element — skip or include — written as two calls.
  2. used[] disappears. We visit each element exactly once, in order; nothing to reuse, nothing to track.
  3. size replaces depth for printing. Output length varies (0 to n); depth still walks elements.

Same skeleton: choose, recurse, undo. Two recursive calls instead of a loop.

The code

int arr[N], chosen[N];
int size = 0;                       // how many elements in the current subset

void subsets(int depth, int n) {
    if (depth == n) {
        printf("{ ");
        for (int i = 0; i < size; i++) printf("%d ", chosen[i]);
        printf("}\n");
        return;
    }
    // choice 1: skip arr[depth]
    subsets(depth + 1, n);

    // choice 2: include arr[depth]
    chosen[size++] = arr[depth];
    subsets(depth + 1, n);
    size--;                         // undo
}

The undo is one line: size--. Same reason as Monday — shared state must be restored before the caller continues.

One more thing about subsets

The structure of subsets — every existing answer can grow into the next answer by one decision — means we can also build them iteratively, no recursion needed.

subsets_iter(arr):
    result ← [ {} ]                            // start with the empty subset

    for each element x in arr:
        new_subsets ← [ ]
        for each subset S in result:
            add (S ∪ {x}) to new_subsets       // include x in this branch
        result ← result + new_subsets          // double the list

    return result


subsets_iter([1, 2, 3])
   →   [ {}, {1}, {2}, {1,2}, {3}, {1,3}, {2,3}, {1,2,3} ]

For each new element, take every existing subset and produce two versions: one with the element, one without. The list doubles. After n elements: 2n subsets.

Two loops, fixed depth, any n.

And the same is true for permutations

On Monday I said iteration cannot enumerate permutations. That was not accurate. The same "build big from small" idea works:

permutations_iter(arr):
    result ← [ "" ]                            // start with one empty permutation

    for each character c in arr:
        next_result ← [ ]
        for each partial p in result:
            for i from 0 to length(p):         // insert c at every position
                add (p[0..i-1] + c + p[i..end]) to next_result
        result ← next_result

    return result

For each new character, take every existing partial permutation and insert the character into every possible position. Three loops, fixed depth, any n.

Same intuition as the n × (n−1)! argument we used on Monday — just coded as a loop instead of a recursion.

Merge Sort

Divide and conquer — the work is in the merge

Problem — solve this first

You have two arrays, each already sorted:

a = [ 2, 5, 8 ]        b = [ 1, 3, 9 ]

Produce one sorted array containing all six values:

[ 1, 2, 3, 5, 8, 9 ]

How? Write the procedure — on paper or in C. Aim for one pass: don't just concatenate and re-sort.

8 minutes. Hint if stuck: you're holding two sorted piles of exam scripts and want one sorted pile. What do you do with your hands?

What you (probably) discovered

Look at the front of each pile. Take the smaller. Repeat.

StepFront of aFront of bTakeOutput
1211[1]
2232[1, 2]
3533[1, 2, 3]
4595[1, 2, 3, 5]
5898[1, 2, 3, 5, 8]
6—99[1, 2, 3, 5, 8, 9]

When one pile empties, the rest of the other pile is already sorted — copy it over.

This is the merge. One pass, two pointers. O(n). You invented it.

Merge — pseudocode

merge(arr, lo, mid, hi):
    create tmp[ ]                              // empty temp array
    i ← lo                                     // pointer into left half
    j ← mid + 1                                // pointer into right half

    while i ≤ mid AND j ≤ hi:                  // both halves have elements
        if arr[i] ≤ arr[j]:
            append arr[i] to tmp;   i ← i + 1
        else:
            append arr[j] to tmp;   j ← j + 1

    while i ≤ mid:                             // leftovers, left half
        append arr[i] to tmp;       i ← i + 1

    while j ≤ hi:                              // leftovers, right half
        append arr[j] to tmp;       j ← j + 1

    for x from 0 to length(tmp) - 1:           // copy back into arr
        arr[lo + x] ← tmp[x]

Use this to trace by hand: at every iteration of the main loop, write down i, j, what was compared, what was appended to tmp.

Now the leap

You can merge two sorted halves.
Can you sort?

Where do sorted halves come from? Sort them. With what? The same function.

Merge sort — pseudocode

mergeSort(arr, lo, hi):
    if lo ≥ hi: return                         // base case: 0 or 1 element
    mid ← (lo + hi) / 2

    mergeSort(arr, lo, mid)                    // trust: left half is sorted
    mergeSort(arr, mid + 1, hi)                // trust: right half is sorted
    merge(arr, lo, mid, hi)                    // combine

The split tree

  SPLIT  (going down)

              [ 5, 2, 8, 1, 9, 3, 7, 4 ]

                     /                 \

         [ 5, 2, 8, 1 ]               [ 9, 3, 7, 4 ]

            /        \                   /        \

       [ 5, 2 ]    [ 8, 1 ]         [ 9, 3 ]    [ 7, 4 ]

         /  \       /  \              /  \       /  \

       [5] [2]    [8] [1]           [9] [3]    [7] [4]      ← base cases



  MERGE  (going up)

       [2,5]      [1,8]              [3,9]      [4,7]

             [1,2,5,8]                     [3,4,7,9]

                       [1, 2, 3, 4, 5, 7, 8, 9]

Splitting is trivial. All the sorting work happens in the merges, on the way back up.

Merge sort — time complexity

Count the work, level by level. Example: n = 8.

level 0:   [ . . . . . . . . ]                 1 merge of 8 elements   →  8 work
level 1:   [ . . . . | . . . . ]               2 merges of 4 each      →  8 work
level 2:   [ . . | . . | . . | . . ]           4 merges of 2 each      →  8 work
level 3:   [.|.|.|.|.|.|.|.]                   base cases, no merge    →  0
                                                                       ─────
                                                            3 × 8 = 24  =  n × log₂ n

Every level does n work in total. Halving from n down to 1 takes log₂ n levels. So total work is n × log₂ n.

Best = Average = Worst = O(n log n). The data does not change the work — the structure is "halve, recurse, merge," and that runs the same way regardless of what's in the array.

Trade-off: predictable speed, paid for with O(n) extra memory for the temp array.

Quick Sort

Same family, work in the opposite place

Flip the idea

Merge sort

Split is trivial (just halve).
Work happens after the recursive calls (merge).

Quick sort

Work happens before the recursive calls (partition).
Combine is trivial (nothing to do).

Partition: pick a pivot value, rearrange so everything smaller is on its left and everything larger on its right. The pivot lands in its final sorted position. Then sort the two sides — recursively.

What partition does

Pick a value from the array — the pivot. Rearrange so:

  • everything ≤ pivot is on its left,
  • everything > pivot is on its right,
  • the pivot itself sits between them — in its final sorted position.
before:   [ 7, 2, 9, 4, 3 ]                pivot = 3   (last element)

after:    [ 2, 3, 9, 4, 7 ]
                ↑
            pivot in its final position
            left of pivot:  all ≤ 3
            right of pivot: all > 3

Return the pivot's index — quick sort needs to know where it landed.

Exercise — design partition

Write a function partition(arr, lo, hi) that satisfies the contract on the previous slide. Use the last element as pivot. Return the pivot's final index.

20 minutes. Test on:

[ 7, 2, 9, 4, 3 ]   →  pivot 3 lands at index 1
[ 5, 8, 1, 6, 4 ]   →  pivot 4 lands at index 1
[ 3, 7, 2, 8, 5 ]   →  pivot 5 lands at index 2

Several valid algorithms exist. The simplest walks the array once with a pointer that tracks the boundary of the "≤ pivot" zone.

Quick sort — pseudocode

quickSort(arr, lo, hi):
    if lo ≥ hi: return                         // base case: 0 or 1 element
    p ← partition(arr, lo, hi)                 // pivot lands at index p

    quickSort(arr, lo, p - 1)                  // sort left of pivot
    quickSort(arr, p + 1, hi)                  // sort right of pivot

The shell is short. partition() does the placement; the two recursive calls handle the two smaller subproblems. No merge step — the pivot is already where it belongs.

Quick sort — time complexity

The cost per level of the recursion is O(n) (one partition over the active range). The depth depends on where pivots land.

CaseComplexityWhat causes it
BestO(n log n)Pivot lands near the middle every time. Tree depth ≈ log₂ n. Balanced.
AverageO(n log n)On random data the partitions are usually reasonably balanced. Small constants — often faster than merge sort in practice.
WorstO(n²)Pivot keeps landing at one end — already-sorted (or reverse-sorted) input with last-element pivot. One side gets nothing, the other gets n - 1. Tree depth ≈ n.
The worst case is real and triggered by bad inputs (already sorted). The next slide shows the standard fixes.

Picking the pivot

Different choices, different exposure to the worst case:

StrategyHowBest / AvgWorstWhen it bites
Last element (today's code) pivot = arr[hi] O(n log n) O(n²) Already-sorted input — every partition leaves n - 1 on one side.
First element pivot = arr[lo] O(n log n) O(n²) Same problem, just triggered by sorted input the other way.
Random Pick a random index r in [lo, hi]; swap arr[r] with arr[hi]; then partition as usual. O(n log n) O(n²) in theory, vanishingly unlikely in practice Almost nothing — the input no longer controls the pivot.
Median of three Take arr[lo], arr[mid], arr[hi]; use their median as the pivot. O(n log n) O(n²) still possible but rare Carefully crafted adversarial inputs. Common in production sorts.
Random and median-of-three each cost a tiny constant per partition — a few extra operations, no change to the O-class. The payoff is robustness: the algorithm's speed stops depending on whether the input happens to be sorted.

The key step to mastermerge (two pointers)partition (pivot placement)
Both are divide-and-conquer: split the problem, trust the recursive calls, do the real work in exactly one place. Merge sort is steady; quick sort is faster on average but can stumble on bad inputs.
