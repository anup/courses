Binary Trees

Where recursion meets data

ES 242(N) · Class 13 · 15 June 2026

From linked list to binary tree

A linked-list node holds a value and a pointer to one next node — the "rest of the list."

Node:
    data
    next  →  pointer to the next Node

What if a node points to two smaller structures instead of one?

The binary tree node

Node:
    data
    left   →  pointer to the left subtree
    right  →  pointer to the right subtree
One pointer → a list. Two pointers → a binary tree. The pattern is the same: a node + smaller versions of the same structure.

The recursive definition

Linked list

A list is either:

  • empty, or
  • a node + the rest of the list

Binary tree

A tree is either:

  • empty, or
  • a root + left tree + right tree

Same skeleton — a base case and a smaller version of itself. The tree just has two smaller versions instead of one.

Drawing a tree

and then writing the code that makes it

A small tree

Root = 10. Left subtree rooted at 20. Right subtree rooted at 30. 40 and 50 are leaves of the left subtree.

One helper to make a node

newNode(data):
    allocate a new Node
    node.data  ← data
    node.left  ← NULL
    node.right ← NULL
    return node

That's the only place memory gets allocated. Every node is born with no children — we hook them up next.

Build the tree, node by node

root ← newNode(10)

root.left  ← newNode(20)
root.right ← newNode(30)

root.left.left  ← newNode(40)
root.left.right ← newNode(50)

Each assignment is one edge in the drawing. root.left means "the left child of the root." root.left.left means "the left child of the left child of the root."

Read the chain of arrows as the path you walk down the tree.

Traversals

Where do we print the root?

The skeleton — three positions for "the work"

traverse(root):
    if root is NULL: return

    // position 1: before left and right

    traverse(root.left)

    // position 2: between left and right

    traverse(root.right)

    // position 3: after left and right

Same recursion, three places to do the work. The choice of position gives the three classic traversals.

The three traversals

Position of printfNameReads as
position 1 (before left)Preorderroot, then left subtree, then right subtree
position 2 (between)Inorderleft subtree, then root, then right subtree
position 3 (after right)Postorderleft subtree, then right subtree, then root
One skeleton. The position of the work line is the only choice. That's all there is to remember.

The three traversals — pseudocode

preorder(root):
    if root is NULL: return
    print root.data           // before
    preorder(root.left)
    preorder(root.right)

inorder(root):
    if root is NULL: return
    inorder(root.left)
    print root.data           // between
    inorder(root.right)

postorder(root):
    if root is NULL: return
    postorder(root.left)
    postorder(root.right)
    print root.data           // after

Computing on trees

Same skeleton. Different return value.

Recap — the traversal skeleton

void traverse(struct Node *root) {
    if (root == NULL) return;
    /* work */
    traverse(root->left);
    /* work */
    traverse(root->right);
    /* work */
}

Three things to compute next — size, height, leaf count — all use this skeleton with one change: they return a value instead of printing.

Size — how many nodes?

If the tree is empty: 0.

Otherwise: 1 (the root) + size of left subtree + size of right subtree.

size(root):
    if root is NULL: return 0
    return 1 + size(root.left) + size(root.right)

Trust the recursive calls: each returns the correct count for its subtree. Combine by adding 1 for the root.

Height — how deep does the tree go?

Height = the length of the longest path from the root down to any leaf.

If the tree is empty: -1 (no nodes, no path).

Otherwise: 1 + max(height of left subtree, height of right subtree).

height(root):
    if root is NULL: return -1
    hl ← height(root.left)
    hr ← height(root.right)
    return 1 + max(hl, hr)

A single-node tree has height 0 (longest path: zero edges). A leaf's children are NULL with height -1; adding 1 gives 0.

Leaf count — how many leaves?

A leaf is a node with no children.

If the tree is empty: 0.
If the node is a leaf (no left, no right): 1.
Otherwise: leaves of left subtree + leaves of right subtree.

leafCount(root):
    if root is NULL: return 0
    if root.left is NULL AND root.right is NULL: return 1
    return leafCount(root.left) + leafCount(root.right)

The unifying pattern

Every recursive tree computation has the same shape:

f(root):
    if root is empty:    return  base-case value
    if root is a leaf:   return  leaf value          (sometimes)

    left_answer  ← f(root.left)              // trust the call
    right_answer ← f(root.right)             // trust the call
    return combine(left_answer, right_answer, root.data)

The function only changes in two places: the base-case value, and the combine step. The recursion writes itself.

Three problems

Same skeleton, three different jobs

Problem 1 — Evaluate an expression tree

Each leaf holds a number. Each internal node holds an operator: +, -, *, /.

Expected result: (3 + 4) * 5 = 35

Walk this tree using all three traversals. What does each one give you? 5 minutes.

What each traversal produces

Walking the same tree three different ways gives three classical notations for the same expression:

TraversalOrder at each nodeOutputNotation
Inorderleft → root → right3 + 4 * 5infix (what we normally write)
Preorderroot → left → right* + 3 4 5prefix
Postorderleft → right → root3 4 + 5 *postfix

All three are valid ways to print the expression. But the question wasn't "print it" — it was "compute its value." Different jobs.

Inorder gives you the form humans read. Postorder gives you the form a computer evaluates without ambiguity. Which is the natural one for computing the value?

Postorder — why it fits evaluation

Stand at the * root. To apply *, we need two numbers — its left and right operands. The left operand is the value of the + subtree, which itself needs 3 and 4 ready first.

An operator can't act until both its operands are ready. That forces the order: process left subtree, process right subtree, then apply the operator at the root. That's postorder. 
evaluate(root):
    if root is a leaf:
        return root.data

    left_value  ← evaluate(root.left)
    right_value ← evaluate(root.right)
    return apply(root.op, left_value, right_value)

Trace on the tree: evaluate(3) → 3; evaluate(4) → 4; evaluate(+) → 3 + 4 = 7; evaluate(5) → 5; evaluate(*) → 7 * 5 = 35.

Day 9: we evaluated postfix expressions using a stack — operands pushed, operators pop two and push the result. The expression tree is the same computation, structured as data instead of a stream. Postorder traversal of the tree is the postfix stack evaluation.

Problem 2 — solution

invert(root):
    if root is NULL: return

    swap root.left and root.right
    invert(root.left)
    invert(root.right)

Same skeleton as a traversal — the "work" line is now swap instead of print, and the work happens at position 1 (could equivalently happen at position 3 — both work).

Notice what makes this small: every node does the same simple job, the recursion takes care of getting to every node.

Problem 3 — Root-to-leaf path sum

Given a binary tree of integers and a target value, is there any path from the root down to a leaf whose values sum to the target?

Target = 15. Is there a root-to-leaf path that sums to 15?

The three root-to-leaf paths: 5→4→11 (sum 20), 5→4→2 (sum 11), 5→8→1 (sum 14). None match 15 — answer is no.

Try target = 14, and also target = 20. 3 minutes to think before turning the page.

Problem 3 — solution

hasPathSum(root, target):
    if root is NULL: return FALSE

    if root is a leaf:
        return (root.data == target)

    remaining ← target - root.data
    return hasPathSum(root.left, remaining)
        OR hasPathSum(root.right, remaining)

For target = 14: at root 5, remaining becomes 9. Down the right side, at node 8, remaining becomes 1. The leaf 1 matches. Return TRUE.

For target = 20: at root 5, remaining becomes 15. Down the left side, at node 4, remaining becomes 11. The leaf 11 matches. Return TRUE.

New pattern: the target flows down the recursion, shrinking as we go. Up to now, our recursive answers were built up — combining left + right + root. Here, information also flows down through the parameter. This is the technique that makes BST search work on Wednesday.
