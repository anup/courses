So what were we doing before?

Every traversal so far followed one branch as deep as it goes, then backtracked and tried the next. That walk has a name: depth-first search.

What we wroteThe walkIt was…
Preordervisit, then recurse left, then rightDFS
Inorderrecurse left, visit, recurse rightDFS
Postorderrecurse left, right, then visitDFS
Validate BSTinorder walk with a running prevDFS
LCA, delete, heightwalk down, combine on the way backDFS
Preorder, inorder, postorder differ in exactly one thing: when you visit the node relative to recursing into the children. All three are DFS. You've been writing DFS all unit without the name.

DFS and BFS are the same code

Write DFS without recursion, and the only difference from BFS is the container you pull from.

DFS — a stack (LIFO)

PUSH(S, root)
while S not empty:
    node ← POP(S)
    visit(node)
    PUSH(S, node.right)
    PUSH(S, node.left)

Push right first, so left comes off the top next — left-to-right preorder.

BFS — a queue (FIFO)

ENQUEUE(Q, root)
while Q not empty:
    node ← DEQUEUE(Q)
    visit(node)
    ENQUEUE(Q, node.left)
    ENQUEUE(Q, node.right)

Children join the back, so a level finishes before the next begins.

Stack → go deep. Queue → go wide. Swap one container for the other and the whole behaviour flips. The recursion you've been writing is DFS borrowing the call stack for free.

One thing we never needed

Look back at every walk: we never kept a "visited" set. We never worried about arriving at a node twice.

We could skip it because a tree has exactly one path from the root to any node. There's no way back to where you've already been.

A graph can have cycles. Now you can arrive twice — and a plain DFS or BFS would loop forever. The fix is one addition: a visited set.
You've already kept one — on the 2D grid, where you could step back onto a cell you'd seen. A grid is just a graph with cycles. Wednesday: graphs. Same two walks — DFS with a stack, BFS with a queue — the same code, plus one guard against going in circles.
