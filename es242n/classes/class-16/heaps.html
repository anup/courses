Heaps

A binary tree with two extra rules — and an array layout for free

ES 242(N) · Class 16 · 22 June 2026

From queues to priority queues

Queues we've seen are FIFO. Many systems need priority instead — whoever ranks highest leaves next, regardless of arrival order:

Money cuts the line

  • Airport boarding — ticket class
  • Theme park fast-pass — paid lane
  • Premium support — VIP tier

Need cuts the line

  • Hospital ER — severity of injury
  • OS scheduler — system task > background
  • Network QoS — voice/video > bulk
Same waiting-line shape, different rule for who leaves next. This is a priority queue.

A concrete case — the OS ready queue

Dozens of processes want the CPU at once. The OS keeps a ready queue, ordered by priority, not by arrival.

What the priority looks like

  • Real-time — interrupt handlers, audio codec, kernel threads
  • Interactive — your typing, the foreground app
  • Background — file indexer, software updates, GC threads

Priority isn't fixed

  • Aging: a process that has waited long is boosted up (prevents starvation)
  • I/O boost: a process that just finished waiting on disk/network gets a short bump (keeps the UI feeling snappy)
  • Demotion: a CPU hog gets moved down so others get a turn
Every context switch the kernel does extract_max(). A new process becomes ready: insert(). The aging tick updates priorities. This is a priority queue. The I/O scheduler, packet scheduler, and several other kernel queues share the same shape.

What does the priority queue need?

Three operations on the data structure:

insert(x)        // add an element with priority x
peek_max()       // look at the highest-priority element
extract_max()    // remove and return the highest-priority element

How would we solve it with what we already know?

Before we invent anything new — with the structures we've already studied (arrays, sorted arrays, balanced BSTs), what costs do we get for each operation?

Structureinsertpeek_maxextract_max
Unsorted array O(1) O(n) O(n)
Sorted array (ascending — max at end) O(n) O(1) O(1)
Balanced BST O(log n) O(log n) O(log n)

Unsorted: insert is free but every extract is a linear scan. Sorted: opposite trade-off. Neither is balanced across the three.

Balanced BST gets all three to O(log n) — but at a steep cost. Rotations to keep it balanced, and a lot of capability we don't need here: search by key, inorder traversal, range queries, kth-smallest.

The question: can we hit O(log n) on all three with a simpler structure — one designed only for the priority queue, with no extra machinery and no unused capability?

What is a heap?

Yes — and the structure is called a heap.

A heap is a binary tree — same node concept as before (data and two children). It is not a BST. The BST imposed one ordering rule across the whole tree. A heap imposes a different pair of rules:

  1. Shape rule: the tree is complete — every level except possibly the last is full, and the last level fills left to right.
  2. Order rule: every parent dominates both its children. In a max-heap, parent ≥ children. In a min-heap, parent ≤ children.

Both rules must hold. That's the whole definition.

A max-heap. Shape: complete. Order: every parent ≥ both children.

Different rules from a BST → different operations get fast. We'll examine each rule, then see what they buy us.

Part 1

A closer look at each rule

Rule 1 — the shape rule (complete binary tree)

A complete binary tree.

A binary tree is complete when:

  • Every level except possibly the last is completely full.
  • The last level fills left to right, with no gaps.

So in the tree to the left: A's level is full (just A). B/C's level is full. The last level — D, E, F — fills left to right, with the rightmost slot still empty.

Completeness is a shape constraint. It says nothing about values yet. We'll add the value rule next.

Rule 2 — the order rule (parent dominates)

The order rule for a max-heap:

Every parent's value is greater than or equal to both of its children's values.

Each parent ≥ its children: 50 ≥ 30, 40 · 30 ≥ 10, 20 · 40 ≥ 35, 25

What the rule does NOT say:

  • Nothing about siblings. Left and right children can be in any order relative to each other.
  • Nothing across cousins. A node in the left subtree can be larger than a node in the right subtree.

This is much weaker than the BST rule, which constrained relationships across the whole tree.

A min-heap is the same idea, flipped: every parent ≤ both children. Root is the smallest.

The two rules together

Shape rule

The tree is complete — full at every level except possibly the last, which fills left to right.

Order rule

Every parent dominates its children — ≥ for max-heap, ≤ for min-heap.

Both rules must hold. A tree with the order rule but not the shape rule is not a heap. A complete tree with random values is not a heap.

Coming up: the shape rule isn't just a constraint — it's the reason we can store the entire tree in an array, without any pointers.

Because the tree is complete, every index from 0 to n−1 corresponds to exactly one node — no gaps. The level-order numbering fits the array indices perfectly.

Where do parent and children sit?

Take any node at index i. With completeness in our pocket, we can read its children's and parent's indices straight off the level structure.

The indices, level by level:

  • Level 0: 1 node  →  index 0
  • Level 1: 2 nodes  →  indices 1, 2
  • Level 2: 4 nodes  →  indices 3, 4, 5, 6
  • Level 3: 8 nodes  →  indices 7, 8, 9, 10, 11, 12, 13, 14

Each level packs twice as many nodes as the one above. So node i on one level lands at a precise offset on the next.

Children of 1: 3, 4. Children of 5: 11, 12.

Part 3

Insert

Where does a new value go?

To keep the shape rule, the only valid place for a new node is the next free slot: the last position on the last level, filling left to right.

In the array: that's just arr[n] — append at the end.

But the new value might break the order rule. If it ends up bigger than its parent (max-heap), we're not done.

Two-step pattern: place at the end (fixes shape), then swim up (fixes order).

Swim up — restoring the order rule

The new node may be larger than its parent. If so, swap them. Then check the new parent. Keep going until either (a) the parent is larger, or (b) we reach the root.

Only the path from the new node to the root can have violations — every other subtree is unchanged. So one walk up is enough to fix everything.

This is sometimes called swim, sift-up, bubble-up, or heapify-up. Same operation under different names.

Insert — trace

Start with the heap [50, 30, 40, 10, 20, 35, 25]. Insert 45.

  1. Append 45 at index 7.
    [50,30,40,10,20,35,25,45]
  2. Parent (i=3) is 10. 45 > 10 → swap.
    [50,30,40,45,20,35,25,10]
  3. Parent (i=1) is 30. 45 > 30 → swap.
    [50,45,40,30,20,35,25,10]
  4. Parent (i=0) is 50. 45 < 50 → stop.

Walked 3 levels — log₂(8). Done.

Insert — pseudocode

INSERT(heap, value):
    heap.append(value)                          // shape rule
    SWIM(heap, length(heap) − 1)                // order rule

SWIM(heap, i):
    while i > 0:
        parent ← (i − 1) / 2
        if heap[i] > heap[parent]:
            swap(heap[i], heap[parent])
            i ← parent
        else:
            return                              // order rule satisfied

Two phases, both small. The append fixes shape. The swim fixes order along one path to the root.

Insert — complexity

What's the worst-case cost of insert?

Swim walks up one level per swap. Worst case: from the bottom level all the way to the root.

Number of levels = O(log n) — because the tree is complete, its height is exactly ⌊log₂ n⌋.

So insert is O(log n). This is one of the heap's two main payoffs.

Compare BST: insert is O(height), which can be O(log n) only if the tree happens to stay balanced. Heap forces balance through the shape rule. There is no "degenerate heap."

Part 4

Extract max — remove the root

Extract max — how it works

The order rule puts the largest value at the root. So removing the max means removing the root. But the resulting tree still has to satisfy both rules.

How?

Three steps:

  1. Save the root's value as the return value.
  2. Move the last element to the root, then shrink the array by one. Shape rule restored; order rule now broken at the root.
  3. Sink down. Swap with the larger of the two children. Then check the new position. Stop when both children are ≤ this node, or we reach a leaf.

Same pattern as insert: a structural fix that breaks order, then walk in one direction to fix it.

Sink down — restoring the order rule

The node we just placed at the root may be smaller than its children. If so, swap it with the larger of the two children. Then check the new position. Keep going until either (a) both children are ≤ this node, or (b) we reach a leaf.

Why the larger child? Swapping with the smaller one would put a small value above a bigger one on the other side — breaking the order rule there.

Only the path the sinking node travels can have violations — every other subtree is unchanged. So one walk down is enough to fix everything.

This is sometimes called sink, sift-down, bubble-down, or heapify-down. Same operation under different names. It's the mirror of swim from insert: same idea, opposite direction.

Extract max — trace

Start with the heap [50, 45, 40, 30, 20, 35, 25, 10]. Extract the max.

  1. Save the root → return 50.
  2. Move last (10) to root, shrink.
    [10,45,40,30,20,35,25]
  3. Children of i=0: 45, 40. Max = 45. 10 < 45 → swap.
    [45,10,40,30,20,35,25]
  4. Children of i=1: 30, 20. Max = 30. 10 < 30 → swap.
    [45,30,40,10,20,35,25]
  5. i=3 is a leaf (children at 7, 8 — out of bounds). Stop.

Returned 50. Walked log₂(7) ≈ 3 levels.

Extract max — pseudocode

EXTRACT-MAX(heap):
    max ← heap[0]
    heap[0] ← heap[length(heap) − 1]
    heap.pop()                                   // shrink by one
    SINK(heap, 0)
    return max

SINK(heap, i):
    n ← length(heap)
    while true:
        left  ← 2*i + 1
        right ← 2*i + 2
        largest ← i
        if left  < n and heap[left]  > heap[largest]: largest ← left
        if right < n and heap[right] > heap[largest]: largest ← right
        if largest = i: return                   // order rule satisfied
        swap(heap[i], heap[largest])
        i ← largest

Extract max — complexity

What's the worst-case cost of extract-max?

Sink walks down one level per swap. Worst case: from the root all the way to a leaf.

Number of levels = O(log n), same reason as insert: completeness fixes the height.

So extract-max is O(log n). With insert also O(log n), we have a structure where the two most useful operations on "the biggest value" cost a log walk.

Part 5

Heapify — building a heap from scratch

The problem

You're given an arbitrary array of n values. Turn it into a valid max-heap, in place.

How fast can we do it?

The obvious way: call INSERT n times, starting from an empty heap. Each insert is O(log n), so total is O(n log n). Correct, but not the best we can do.

The smarter way: sink each non-leaf node, starting from the bottom and working up. Total time: O(n).

Bottom-up heapify

Key idea: leaves are already heaps of size 1. They trivially satisfy the order rule. The only nodes that might violate it are the internal (non-leaf) nodes.

Start from the last non-leaf node and walk backwards (down to index 0). At each one, SINK it.

By the time we sink a node, both its subtrees are already valid heaps — so a single sink restores the property for the whole subtree rooted here.

HEAPIFY(arr):
    n ← length(arr)
    for i from (n / 2) − 1 down to 0:
        SINK(arr, i)

n/2 − 1 is the index of the last non-leaf node. Anything beyond that is a leaf (children indices ≥ n).

Heapify — trace

Start with [10, 20, 50, 30, 40, 35, 25]. Last non-leaf is at index 2.

  1. i=2 (50). Children 35, 25. 50 is largest → no swap.
  2. i=1 (20). Children 30, 40. Max = 40. 20 < 40 → swap.
    [10,40,50,30,20,35,25]
  3. i=0 (10). Children 40, 50. Max = 50. 10 < 50 → swap.
    [50,40,10,30,20,35,25]
  4. Continue sinking 10. At i=2, children 35, 25. Max = 35. 10 < 35 → swap.
    [50,40,35,30,20,10,25]
  5. i=5 leaf. Done.

Result: [50,40,35,30,20,10,25] — valid max-heap.

Why heapify is O(n), not O(n log n)

The naive bound: n/2 sinks, each up to O(log n) → O(n log n). But this is loose.

The cost of a sink depends on how deep that node's subtree goes — not the full tree's height. Most nodes have very shallow subtrees underneath them.

Level from bottomNodesMax sink work eachTotal work at this level
Leaves (h=0)≈ n/200
h = 1≈ n/4≤ 1≈ n/4
h = 2≈ n/8≤ 2≈ 2n/8 = n/4
h = 3≈ n/16≤ 3≈ 3n/16
…………

Total = n · ∑ h / 2h+1 for h = 0, 1, 2, …   The infinite sum ∑ h / 2h equals 2, a constant. So total ≈ n. Heapify is O(n).

Intuition: inserting one at a time pays the worst-case log cost n times. Heapify shifts the work to where it's cheap — the bottom level has the most nodes but each one barely sinks.

Part 6

Where heaps show up

Top-K problems

"Given a stream of values, what are the top-K largest so far?"

Naive: keep them all sorted. Expensive on every insert.

Smart: maintain a min-heap of size K. For each new value:

  • If the heap has fewer than K elements, insert it.
  • Else if the value is larger than the heap's min (the root), replace the root with this value and sink down.
  • Else ignore it — it's not in the top K.

Each step is O(log K), independent of n. The heap stays at size K throughout.

Heap sort

We have heapify in O(n) and extract-max in O(log n). Can we use them to sort an array? At what cost?

A sorting algorithm built straight out of heap operations:

  1. Heapify the input array — O(n).
  2. Repeatedly extract the max. Place each extracted value at the end of the array (the slot just vacated by the heap shrinking).

After n extracts, the array is sorted in ascending order. Total time: O(n) + n × O(log n) = O(n log n), in place.

Same asymptotic as merge sort, in-place like quick sort — without quick sort's O(n²) worst case.

STL priority queues — different defaults across languages

C++ STL's std::priority_queue and Java's PriorityQueue are both heap-backed.

Quick: max-heap or min-heap by default?

Trick question. It depends on the language.

C++

  • priority_queue<int> = max-heap by default.
  • top() = the max.
  • push(x) = insert; pop() = extract.
  • For a min-heap: pass greater<int> to flip the ordering.

Java

  • PriorityQueue<Integer> = min-heap by default (the opposite of C++!).
  • peek() = the min; poll() = extract.
  • For a max-heap: pass Comparator.reverseOrder() to flip the ordering.
Same data structure, opposite defaults across languages. Read the docs before you reach for the root.

What we built today

  • Heap = binary tree + two rules (shape: complete; order: parent dominates).
  • The shape rule lets us store the whole tree as an array, with index arithmetic instead of pointers.
  • Insert: append at end, swim up. O(log n).
  • Extract max: swap root with last, shrink, sink down. O(log n).
  • Heapify: bottom-up sinks. O(n).
  • Priority queues, top-K, heap sort — all built on top of these.
Heaps and BSTs are both "binary tree + rules" — but the rules they pick are different. BST's ordering gives sorted traversal; heap's shape + ordering gives a fast priority queue.
