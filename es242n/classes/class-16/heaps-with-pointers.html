Two ways to label a node

Every node has an array index (0-indexed, as in class) and a level-order count (1-indexed). The pointer trick ahead uses the count — its binary form spells the route from the root.

index = count − 1

Pointers

That search is the only hard part of a pointer heap. Swim and sink don't change.

The shape rule does the work

A heap is complete: positions 1 … n are filled densely, left to right. So a route computed from a position number is guaranteed to land on a real node — it can't dead-end.

Heap — positions 1..6 filled

Route to 6: 1 → 3 → 6 ✓

Sparse tree — positions missing

Route to 6: 1 → 3 → ? dead-end ✗

A heap fills positions densely; a BST (or any non-complete tree) can have gaps. The bit-path trick belongs to the heap because completeness is what keeps the route alive.

The heap keeps one field: its size n. From n we find both.

Approach 1 — the queue

"Next free slot in level order" sounds like a queue (BFS).

Dequeue nodes level by level. The first node we dequeue that is missing a child — that's where the new node goes.

Approach 1 — pseudocode

INSERT-QUEUE(root, value):
    newNode ← Node(value)
    if root = NIL: return newNode

    Q ← empty queue;  ENQUEUE(Q, root)
    while Q not empty:
        node ← DEQUEUE(Q)
        if node.left = NIL:                    // first gap
            node.left ← newNode;  newNode.parent ← node;  break
        ENQUEUE(Q, node.left)
        if node.right = NIL:                   // first gap
            node.right ← newNode; newNode.parent ← node;  break
        ENQUEUE(Q, node.right)

    SWIM-UP(newNode)
    return root

...but the queue scans the whole tree

BFS visits all 7 nodes before reaching the gap.

O(n)

The gap sits at the bottom, so BFS visits almost every node before finding it.

Swim/sink is only O(log n) — so the search dominates, and a heap op drops to O(n). That defeats the heap.

Approach 2 — the position is a route

In binary, ×2 shifts left and appends a 0; 2k+1 appends a 1. So each step down the tree writes one more bit — and the bits, top to bottom (after dropping the leading 1), are the route. 0 = left, 1 = right.

Two steps — we computed the route from the number. O(log n).

Go to any position

// node at level-order position k  (root is position 1)
NODE-AT(root, k):
    // k is the node's count: the kth node, root = 1
    path ← binary(k) without the leading 1      // k = 6  →  "10"
    node ← root
    for bit in path, most-significant first:
        node ← (bit = 0) ? node.left : node.right
    return node

Path length = number of bits = ⌊log₂ k⌋ → O(log n). (In code: shifts/masks, or halve k onto a stack — no string needed.)

Insert with the number

INSERT-NUM(root, value):
    n ← n + 1                              // new node is the nth
    if n = 1: root ← Node(value); return root

    parent ← NODE-AT(root, n / 2)          // parent = ⌊n/2⌋-th node
    newNode ← Node(value); newNode.parent ← parent
    if n is even: parent.left  ← newNode   // even k → left
    else:         parent.right ← newNode   // odd  k → right

    SWIM-UP(newNode)
    return root

Before: n = 5

After: n = 6 (pos 6 added)

navigate O(log n) + swim O(log n) = insert O(log n). The cost we wanted.

Extract-max with the number

EXTRACT-MAX-NUM(root):
    if n = 0: error "empty"
    maxVal ← root.value
    if n = 1: root ← NIL; n ← 0; return maxVal

    last ← NODE-AT(root, n)                 // the last node
    root.value ← last.value                 // move its value to root
    if n is even: last.parent.left  ← NIL   // detach last
    else:         last.parent.right ← NIL
    n ← n − 1

    SINK-DOWN(root)
    return maxVal

Before: n = 6 (remove pos 6)

After: n = 5

find last O(log n) + sink O(log n) = extract O(log n).

Swim & sink — unchanged

Swim up (after insert)

SWIM-UP(node):
  while node.parent ≠ NIL
        and node.value > node.parent.value:
    swap(node.value, node.parent.value)
    node ← node.parent

Sink down (after extract)

SINK-DOWN(node):
  loop:
    big ← node
    if node.left  ≠ NIL and node.left.value  > big.value:  big ← node.left
    if node.right ≠ NIL and node.right.value > big.value:  big ← node.right
    if big = node: return
    swap(node.value, big.value);  node ← big

Only the value travels; the node objects stay put. Swim moves upward — that's why each node carries a parent pointer (or you keep the descent path on a stack).

The verdict

find the position by…howcost
Queue (BFS)visit positions level by levelO(n)
Number (bits)compute the route from nO(log n)
Array indexposition n is the indexO(1)
The bits of a node's count are its path — and the array stores that node at index count − 1. Pointers re-derive the route one bit at a time; the array hands you the same answer for free. Different ways of asking; the array just answers fastest.

Don't search for the position —
the number already says where it is.
The array just says it out loud.

Arrays win for the binary heap we built. Pointer-based heaps still earn their keep elsewhere — leftist / skew heaps (fast merge), Fibonacci heaps (Dijkstra's decrease-key), persistent heaps. Different shape, different tradeoffs.
