22 June 2026: Heaps

A heap is a binary tree with two extra rules — the shape rule (complete) and the order rule (parent dominates children). The shape rule has a surprising consequence: the whole tree can be stored as an array, with index arithmetic in place of pointers. From there, insert and delete-root both cost a single walk up or down: O(log n).

Open

Opening — concluding binary search trees

Four slides to close the tree unit. The traversals already written — inorder, validate, LCA, delete — were all depth-first search; name it. Then its sibling, breadth-first search: level order with a queue, shown live on the canonical tree. The two are the same code with the container swapped — a stack for DFS, a queue for BFS. Close on the one thing trees let us skip, a visited set, because that is exactly what graphs add on Wednesday.

S1

Session 1 — The two rules and the array trick

Recall the binary tree node from Day 13. Add the shape rule: complete binary tree, filled left to right at the last level. Then the order rule: every parent dominates both children (max-heap or min-heap). The two rules together are what we call a heap. Show "which of these are heaps?" trees to lock both rules in.

Then the consequence of the shape rule: because the tree is complete, we can number nodes level-order and store them in a contiguous array. The parent at (i−1)/2, children at 2i+1 and 2i+2. No pointers, no malloc — just an array and three formulas. Callback to Day 5's data[]/nxt[] linked-list-in-an-array: same theme, sharper version.

S2

Session 2 — Operations and what they're good for

Insert: append at the end (fixes shape), then swim up to restore order. Worked trace inserting 45 into the canonical heap, then pseudocode and complexity. Extract max: save the root, move the last element to the root, shrink, then sink down to restore order. Worked trace, pseudocode, complexity. Both are O(log n) — because completeness fixes the height at log n. There is no "degenerate heap."

Build a heap from an arbitrary array — heapify. The naive insert-each gives O(n log n); the smarter bottom-up sink gives O(n). Short intuition for why the analysis collapses (most sinks are short because they're near the leaves). Close with where heaps actually live: priority queues, top-K with a size-K min-heap, heap sort, and the dhokha angle that std::priority_queue defaults to a max-heap in C++ but PriorityQueue defaults to a min-heap in Java.

In-class exercises

  • Trace inserting 45 into [50, 30, 40, 10, 20, 35, 25] step by step.
  • Trace extracting the max from [50, 45, 40, 30, 20, 35, 25, 10] step by step.
