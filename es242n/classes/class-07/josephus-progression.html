One Problem, Three Implementations

Josephus — finally, we code it

ES 242(N) · Day 7 · 25 May 2026

Before we begin — a tiny puzzle

Seven people stand in a circle. Starting from position 1, count 1, 2, 3 — whoever lands on 3 leaves. Resume counting from the next person.

Who is the last one standing?

Turn to your neighbour. 60 seconds. Pen, paper, fingers — no laptops yet.

Survivor: position 4

Order of elimination: 3 → 6 → 2 → 7 → 5 → 1. Last standing: 4.

If you guessed 4, you'd have made it out of the cave. If you didn't — well, that's what the next 80 minutes are for.

Wait — haven't we done this?

We sketched Josephus when we did arrays. We waved at it again with linked lists. Both times we said "we'll code it properly soon."

Soon is today.

Plan: solve Josephus three times. Once with an array. Once with pointers over the same array. Once with a circular linked list. Compare what changed.

What you'll code

Input: n and k.
Process: n people, numbered 1 to n, in a circle. Count k; the k-th leaves. Repeat.
Output: the 1-indexed survivor.

Check your code on: n = 7, k = 3 → 4.   Stretch: n = 41, k = 3 → 31.

41 and 3 are the original numbers from the cave. If your program prints 31, congratulations — you'd have lived to write a book.

Round 1 — Array

Mark the dead. Keep walking past them.

Approach — array

Use a flag array. 1 = alive, 0 = out. When someone is eliminated, don't shift anything — just flip their flag.

Hints

  • Array of size n for the flags.
  • Walk with an index. Wrap using (idx + 1) % n.
  • Skip indices whose flag is 0. They don't count toward the k.
  • Stop when exactly one flag is still 1. Print that position.
No shifting, no resizing. Just a circular walk past a growing graveyard of zeros.

Your turn — array

Read n, k. Print the 1-indexed survivor.
Submit on judge.csbasics.in.
Verify: 7 3 → 4.   Stretch: 41 3 → 31.
If 7 3 prints 4, you've out-survived six imaginary people. Modest, but a start.

Round 2 — Pointer

Same array. Same memory. Different syntax.

Approach — pointer

alive[i] is just sugar for *(alive + i). Same array; walk it with a pointer instead of an index.

Hints

  • Two anchors: start at the first element, end just past the last.
  • One moving pointer cur. Advance with cur++.
  • Wrap with if (cur == end) cur = start; — that's the circle.
  • Read and write flags as *cur. No [] allowed.
  • Survivor's index at the end: cur - start.
The algorithm hasn't changed. Only the syntax for "where am I."

Your turn — pointer

Same logic, pointer arithmetic only. No alive[i].
Output must match your array version, line for line.
If [] feels easy and *cur feels foreign — that gap is exactly what this exercise exists to close.

Round 3 — Circular linked list

Don't mark the dead. Unlink them.

Approach — circular linked list

Build a circular linked list of n nodes. Each next points to the next person. The last node's next loops back to the first.

Elimination is no longer "flip a flag." It's prev->next = cur->next, then free(cur).

Hints

  • Build one node at a time with malloc. Keep a head and a tail.
  • After the last node: tail->next = head. That one line is what makes it a circle.
  • From cur, walk k − 1 steps. Keep a prev the whole time.
  • Unlink: prev->next = cur->next. Save cur in a temp so you can free it.
  • Move cur to cur->next. Stop when cur->next == cur.
Forgetting tail->next = head is the classic bug. The program walks off the end and crashes spectacularly. Now you know.

Your turn — linked list

Build a circular list with malloc. Eliminate by unlinking and free-ing.
Same survivor as your array and pointer versions — verify.
Three programs, one answer. If two of them disagree, one of them is lying. Welcome to debugging.

So which one was best?

Three implementations. Time to ask what each one cost.

What did the array cost?

Late in the game, most slots are dead, but you still walk through them to find the next live one. You skim over a lot of zeros to count each living person.

Pointer version: identical cost. Pointer arithmetic is not magic — it accesses the same memory.

Same work in different syntax is not faster. It just makes you fluent in C.

What did the linked list cost?

Each round, walk exactly k − 1 nodes, then unlink. No stepping over dead cells — there are none. You removed them.

The win isn't asymptotic — it's structural. The data structure has the same shape as the problem (a circle). The code reads almost line-for-line like the problem statement.

Pick the structure whose shape matches the problem. Josephus is a circle. The linked list literally is one. The array had to fake it.

Big-O — the formal version

The growth families you just felt — flat, linear, quadratic — get named in the Time Complexity deck:

  • Linear search → O(n)
  • Binary search → O(log n)
  • Bubble sort → O(n²)
Same families, formal names, and why we get to drop constants and lower-order terms.

Wrap problem — reverse a sequence, three ways

Same drill, different problem. Reverse n integers using:
  1. An array — swap from both ends, with indices.
  2. The same swap — with pointer arithmetic.
  3. A pointer-based linked list — return a new head pointing the other way.

The linked-list version is the only one where you cannot "swap from both ends." You have to walk forward. That's the data structure forcing the algorithm's hand.

What we did today

  • Took a problem we'd only sketched — Josephus — and finally coded it.
  • Coded it three times. Same algorithm, different data structure.
  • Saw why the linked list reads like the problem and the array doesn't.
  • Met Big-O informally; the formal names live in the time-complexity deck.
Take-away: when picking a data structure, ask "does this look like the problem?" If yes, the code mostly writes itself.
