Strings in C

Characters, Arrays, and the Null Terminator

Introduction to DSA (C Programming)

Strings in Python — Easy

name = "Alice"
print(name)            # Alice
print(len(name))       # 5
print(name[0])         # A
print(name + "!")      # Alice!
print(name == "Alice") # True
Python's str is a full-featured built-in type. C has no string type at all.

Part 1: The char Type

A character is just a number

The char Type

C has a type for a single character — 1 byte:

char c = 'A';       // single quotes for a single character
printf("%c\n", c);  // A   ← print as character
printf("%d\n", c);  // 65  ← print as integer
A char is just a small integer. 'A' is simply the number 65. The single quotes tell the compiler to look up the ASCII value.

char is an Integer — Arithmetic Works

char c = 'A';

printf("%c\n", c + 1);   // B  (65+1 = 66 = 'B')
printf("%c\n", c + 2);   // C

// Range checks using char comparison
if (c >= 'A' && c <= 'Z') {
    printf("uppercase\n");
}

// Convert uppercase to lowercase
char lower = c + 32;     // 'A'(65) + 32 = 'a'(97)
printf("%c\n", lower);   // a

This works because characters are just numbers. ASCII defines the mapping.

ASCII: The Mapping

CharValue CharValue CharValue
'A'65 'a'97 '0'48
'B'66 'b'98 '9'57
'Z'90 'z'122 ' '32
'\n'10 '\0'0 '\t'9
'a' - 'A' = 32 always. Uppercase → lowercase: c + 32. Lowercase → uppercase: c - 32.

Part 2: Strings are Arrays

There is no string type in C

A String is a char Array

There is no string type in C. A string is an array of char:

char name[6] = {'A', 'l', 'i', 'c', 'e', '\0'};

"Alice" has 5 characters but needs 6 bytes — the last one is always '\0'

The Null Terminator '\0'

'\0' is the number 0. It signals "this is where the string ends." Without it, C has no way to know where to stop. 
// Without '\0' — printf doesn't know where to stop
char broken[5] = {'A', 'l', 'i', 'c', 'e'};  // no '\0'!
printf("%s\n", broken);  // "Alice" then garbage bytes ⚠️

// With '\0' — printf stops correctly
char name[6] = {'A', 'l', 'i', 'c', 'e', '\0'};
printf("%s\n", name);    // Alice ✓

This is also why every string function loops until it hits '\0'.

The Easier Way to Initialize

String literals add '\0' automatically:

// All three are identical:
char name[6] = {'A', 'l', 'i', 'c', 'e', '\0'};
char name[6] = "Alice";   // compiler adds '\0'
char name[]  = "Alice";   // compiler counts size too → 6
Double quotes "..." are for strings. Single quotes '...' are for a single character. They are not interchangeable. 
char c  = 'A';   // ✓ one char, value 65
char c  = "A";   // ✗ type mismatch — "A" is a char array!

⚠️ String Literals are Read-Only

char array — modifiable ✓

char name[] = "Alice";
name[0] = 'B'; // ✓ OK
// name is now "Blice"

Array lives on the stack — you own it.

char pointer — read-only ✗

char *name = "Alice";
name[0] = 'B'; // ✗ CRASH
// literal is in protected
// read-only memory

Pointer into the program's read-only data segment.

Rule: If you need to modify a string, always declare it as char name[], not char *name.

Part 3: Reading and Printing

printf and scanf

Printing Strings: %s

char name[] = "Alice";

printf("%s\n", name);          // Alice
printf("Hello, %s!\n", name);  // Hello, Alice!

// Under the hood — printf walks until '\0':
for (int i = 0; name[i] != '\0'; i++) {
    printf("%c", name[i]);
}
printf("\n");

The loop condition name[i] != '\0' is the fundamental pattern for all string operations in C.

Reading a Single Word: scanf("%s")

char name[50];
printf("Enter your name: ");
scanf("%s", name);   // no & needed — array name is already a pointer
printf("Hello, %s\n", name);
Problem: scanf("%s") stops at any whitespace. Type "Alice Smith" — you only get "Alice". "Smith" is left sitting in the input buffer. 
// Input typed: Alice Smith
scanf("%s", name);
printf("%s\n", name); // Alice  ← "Smith" is still in the buffer!

The Leftover '\n' Problem

After scanf("%d"), the Enter key ('\n') stays in the input buffer.

This causes a problem with %c and %[^\n] — but NOT with %s:

This works fine ✓

scanf("%d", &age);
scanf("%s", name);
// %s skips whitespace
// automatically

These have problems ✗

scanf("%d", &age);
scanf("%c", &initial);
// reads '\n' instead!

scanf("%d", &age);
scanf("%[^\n]", name);
// reads empty string!
Why? %s and numeric formats (%d, %f) skip leading whitespace automatically. But %c and %[^\n] do NOT skip — they read immediately.

The Fix: Add a Space

A space before the format specifier tells scanf to skip whitespace first:

int age;
char initial;

scanf("%d", &age);
scanf(" %c", &initial);   // ✓ space skips the '\n'
int age;
char name[50];

scanf("%d", &age);
scanf(" %[^\n]", name);   // ✓ space skips the '\n'
The space in the format string means: "skip all whitespace (spaces, tabs, newlines) before reading". Always use " %c" and " %[^\n]" after reading numbers.

A New Tool: the Scanset %[...]

So far we know %d, %f, %c, %s — each reads a specific type.

%[...] is different: you give scanf a set of characters to match.

scanf("%[aeiou]", s);   // reads only vowels, stops at anything else
scanf("%[0-9]",   s);   // reads only digits, stops at anything else
scanf("%[^\n]",   s);   // reads everything EXCEPT newline

Let's break down %[^\n] symbol by symbol:

SymbolMeaning
%Start of a format specifier — same as in %d, %s
[Open a character set — "match characters from this set"
^Negate the set — "match anything not in this set"
\nNewline — the one character we want to exclude
]Close the character set
%[^\n] = "keep reading any character that is not a newline". It stops the moment the user presses Enter.

Reading a Full Line: the Scanset %[^\n]

To read a full line including spaces, use a scanset:

char name[50];
scanf(" %49[^\n]", name);
How to read %[^\n]:
%[...] — match a set of characters
^ — "not" (invert the set)
\n — newline
Result: "match any character that is not a newline" 
// Input: Alice Smith
scanf(" %49[^\n]", name);
printf("%s\n", name);  // Alice Smith ✓

The 49 is a width limit — reads at most 49 characters, leaving 1 byte for '\0' in a 50-char array. Always set this — without it the scanset has no size limit.

All Three scanf String Options

Format Reads spaces? Stops at Size limit?
scanf("%s", s) No any whitespace ⚠️ none — dangerous
scanf("%49s", s) No any whitespace ✓ 49 chars
scanf(" %49[^\n]", s) Yes '\n' only ✓ 49 chars
Always set a width limit equal to array_size - 1. The -1 leaves room for '\0'.
For a single word: scanf("%49s", s)
For a full line: scanf(" %49[^\n]", s)

Part 4: gets()

A simpler way to read strings

scanf for Full Lines is Complicated

Look at what we need just to read a full line:

scanf(" %49[^\n]", name);

That's a lot to remember just to read a name. There is a simpler function.

What if there was a function that just read a full line — no format strings, no width, just pass it the array?

Meet gets()

One argument. No format string. Reads the whole line including spaces.

char name[50];
gets(name);

// Input: Alice Smith
// name now contains: "Alice Smith\0"
printf("%s\n", name);  // Alice Smith

scanf

scanf(" %49[^\n]", name);
// remember the space
// remember the 49
// remember [^\n]

gets

gets(name);
// reads spaces too
// stops at Enter
// adds '\0' for you
gets() reads a full line into the array and adds '\0' at the end. Clean, simple, readable.

What happens when input is longer than the array

Both keep writing past the end of the array into memory that belongs to something else — other variables, the return address, the OS. This is a buffer overflow.

scanf Can Be Fixed. gets() Cannot.

scanf lets you add a width limit to prevent overflow:

scanf("%4s", name);         // reads at most 4 chars ✓
scanf(" %4[^\n]", name);    // reads at most 4 chars with spaces ✓

gets() has no such option:

gets(name);    // no way to tell it how big name is — ever
There is no way to make gets() safe. This is why it was removed from the C11 standard entirely. It still compiles on most systems but produces a warning. Always use scanf with a width limit instead.

What a Buffer Overflow Actually Destroys

Your function's stack frame — the memory block that holds its local variables — looks like this:

Normal execution

Stack memory:
┌──────────────────┐ ← high address
│  return address  │  where to go after main()
│  other variables │
│  name[4]  'a'    │ ← your array ends here
│  name[3]  'x'    │
│  name[2]  'e'    │
│  name[1]  'l'    │
│  name[0]  'A'    │
└──────────────────┘ ← low address

After gets() overflow

Stack memory:
┌──────────────────┐
│  return address  │ ← 💥 OVERWRITTEN
│  other variables │ ← 💥 OVERWRITTEN
│  name[4]  'a'    │
│  name[3]  'x'    │
│  name[2]  'e'    │
│  name[1]  'l'    │
│  name[0]  'A'    │
└──────────────────┘
When the function returns, it jumps to the overwritten return address — which now points to garbage, or in a real attack, to malicious code injected by the user. This is how buffer overflow exploits work. gets() was removed from the C11 standard entirely.

Part 6: String Functions

#include <string.h>

Standard String Functions

Function What it does Python equivalent
strlen(s) Length (not counting '\0') len(s)
strcpy(dst, src) Copy src into dst dst = src
strcat(dst, src) Append src onto end of dst dst += src
strcmp(s1, s2) Compare two strings s1 == s2
strchr(s, c) Find first occurrence of char c in s
strstr(s, sub) Find substring sub in s

strlen() — and writing our own

char name[] = "Alice";
printf("%zu\n", strlen(name));  // 5  (not 6 — doesn't count '\0')

Let's write it ourselves:

int my_strlen(char s[]) {
    int count = 0;
    while (s[count] != '\0') {
        count++;
    }
    return count;
}
// "Alice" → walks [0][1][2][3][4], hits '\0' at [5] → returns 5
Key insight: Every string function works by walking the array until it hits '\0'. Master this pattern and the rest follows.

strcmp() — The Python Gotcha

You cannot compare strings with == in C. It compares addresses, not contents. 
char s1[] = "hello";
char s2[] = "hello";

if (s1 == s2)            { }  // ✗ WRONG — same address? (no!)
if (strcmp(s1, s2) == 0) { }  // ✓ CORRECT — same contents?

strcmp() returns:

  • 0 — strings are equal
  • < 0 — s1 comes before s2 alphabetically
  • > 0 — s1 comes after s2 alphabetically
strcmp("apple", "apple")  →  0
strcmp("apple", "mango")  →  negative  ('a' < 'm')
strcmp("mango", "apple")  →  positive

Writing our own strcmp()

int my_strcmp(char s1[], char s2[]) {
    int i = 0;
    while (s1[i] != '\0' && s2[i] != '\0') {
        if (s1[i] != s2[i]) {
            return s1[i] - s2[i];  // difference in ASCII values
        }
        i++;
    }
    return s1[i] - s2[i];  // handles different lengths
}
// "abc" vs "abd": walks until [2], returns 'c'-'d' = -1

Comparison is purely by ASCII value — which is why 'A'(65) sorts before 'a'(97), and "Zoo" sorts before "ant".

strcpy() — How Buffer Overflow Happens

int i = 0;
while (src[i] != '\0') {

The '\0' lands one byte past the end of dst. That byte belongs to another variable. You've silently corrupted it — no error, no warning.

strcpy() — The Safe Version

char dst[5];

// Dangerous:
strcpy(dst, "Hello");                       // ⚠️ overflow

// Safe: strncpy limits how many chars are written
strncpy(dst, "Hello", sizeof(dst) - 1);    // copies at most 4 chars
dst[sizeof(dst) - 1] = '\0';               // MUST add '\0' yourself!
// dst is now "Hell\0" — truncated, but safe
Why add '\0' yourself? If source is longer than n, strncpy fills the buffer but does not null-terminate. You must always do it manually.

Writing our own safe copy:

void my_strcpy(char dst[], char src[]) {
    int i = 0;
    while (src[i] != '\0') {
        dst[i] = src[i];
        i++;
    }
    dst[i] = '\0';   // don't forget!
}

If that condition isn't met, you will overflow. Use strncat(dst, src, n) to limit how many characters are appended.

Writing our own strcat()

void my_strcat(char dst[], char src[]) {
    int i = 0, j = 0;

    // Step 1: walk to the end of dst
    while (dst[i] != '\0') i++;

    // Step 2: copy src starting from there
    while (src[j] != '\0') {
        dst[i] = src[j];
        i++; j++;
    }

    // Step 3: null-terminate
    dst[i] = '\0';
}

char greeting[20] = "Hello, ";
my_strcat(greeting, "Alice");
printf("%s\n", greeting);  // Hello, Alice
Step 1 is why strcat gets slower the longer dst is — it always walks the whole string from the beginning to find the end.

All the C String Gotchas

Gotcha Wrong ✗ Right ✓
Comparing strings s1 == s2 strcmp(s1, s2) == 0
Reading with spaces scanf("%s", s) scanf(" %49[^\n]", s)
Modifiable string char *s = "hi" char s[] = "hi"
After scanf("%d") next read gets '\n' add a space: " %s" or " %[^\n]"
Reading a line gets(s) — banned! scanf(" %49[^\n]", s)
Copying safely strcpy(dst, src) strncpy + manual '\0'
Forgetting '\0' garbage output / overflow always null-terminate!

Key Takeaways

  1. char is just a small integer — ASCII is the mapping
  2. Strings are char arrays — there is no string type in C
  3. Every string ends with '\0' — forget it and things break
  4. Use scanf(" %49[^\n]", s) to read full lines safely via scanf
  5. Use strcmp(), never ==, to compare strings
  6. strcpy and strcat can overflow — always check sizes first
  7. gets() is banned — it has no size limit at all
