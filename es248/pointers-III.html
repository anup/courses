Operator Precedence — You Already Know This

Some operators bind tighter than others. You learned this in school:

2 + 3 * 4      // = 14, not 20  (* binds tighter than +)
(2 + 3) * 4    // = 20          (parentheses override everything)

C has the same idea — every operator has a precedence level. When two operators compete for the same operand, the higher-precedence one wins.

Parentheses always win. They are the highest-precedence construct in C — they let you override any default binding order.

Precedence Mistakes That Bite Everyone

Bitwise vs equality

if (x & 3 == 0)
// → x & (3==0) → x&0
// always false!

if ((x & 3) == 0) // ✓

== binds tighter than &

&& beats ||

a==1 || a==2 && b==3
// C reads:
// a==1 || (a==2 && b==3)

// always be explicit

&& binds tighter than ||

*p++ vs (*p)++

*p++    // *(p++)
        // ptr advances

(*p)++  // ptr stays
        // value increments

postfix ++ beats unary *

When in doubt, add parentheses. They cost nothing and make intent clear.

Associativity — When Precedence Is Equal

When two operators have the same precedence, associativity decides which groups first.

Left → Right (most operators)

a - b - c
// = (a - b) - c

10 - 3 - 2
// = (10 - 3) - 2 = 5
// NOT 10 - (3 - 2) = 9

Right → Left (assignment)

a = b = 5
// = a = (b = 5)
// b gets 5, then a gets 5

int x, y, z;
x = y = z = 0; // all set to 0
Unary operators also associate right-to-left: **p reads as *(*p) — dereference from the outside in.

Why Are We Looking at Precedence?

In Pointers II, when working with 2D arrays, you hit this pair of declarations:

int *p[4]

array of 4 pointers — [] won

int (*p)[4]

pointer to an array of 4 — * won (forced by parens)

Moving the * inside parentheses completely changes the meaning. Why? Because the same precedence table governs declarations, not just expressions.

In int *p[4], the compiler sees [] and * competing for p. Since [] ranks higher, it wins. Adding (*p) forces * to bind first — same as parentheses overriding precedence in arithmetic.

There is no special rule for pointer declarations. It is the same one rule, applied everywhere.

The Right-Left Rule

Not a new rule — a reading procedure for applying the same precedence table to declarations mechanically. For simple cases you can apply the table directly; for complex nested ones you need a systematic walk.

  1. Find the name
  2. Look right — read [] or () if present
  3. Look left — read * if present
  4. Hit )? Jump to its (, repeat from inside
  5. Stop at the base type
The one question at every step:
"What binds to the name first — array [], function (), or pointer *?"

No parens → [] / () win.
Parens around *name → * wins.

One rule. Any declaration. The only question is: did you put parentheses around *name?

A Program in Memory — The Full Picture

You have seen stack and heap. A running program actually has four distinct regions:

high addresses

— free space —
↑ grows up

Text — your compiled code. Fixed size. Read-only. Set at compile time.

Data — global and static variables you gave an initial value.

BSS — global and static variables with no initial value. The OS zeroes this region at startup — it takes no space in the binary file.

Heap — you already know this.

Stack — you already know this.

What Lives Where?

int g = 5;          // Data   — initialised global
int h;              // BSS    — uninitialised global (zero at start)
static int s = 3;   // Data   — initialised static

void foo() {
    int x = 10;     // Stack  — local variable
    static int c;   // BSS    — static local (persists across calls)

    int *p = malloc(sizeof(int));
    //  p itself    → Stack  (the pointer variable)
    //  *p          → Heap   (the memory p points to)
}
BSS explained: BSS stands for Block Started by Symbol — a name from 1950s assembly. The key insight: the OS zero-fills it at load time, so zeros are never stored in the binary. A program with a large uninitialised array has a small executable file.

Functions Live in Memory Too

Variables have addresses. So do functions — their compiled instructions sit in the text segment at a fixed address.

int add(int a, int b) { return a + b; }
int mul(int a, int b) { return a * b; }

printf("%p\n", (void*)add);   // 0x1040 — address of add's machine code
printf("%p\n", (void*)mul);   // 0x1080 — different address

add  (no parentheses)

The address of the function's code — does not call it. Same as writing an array name without [i]: you get the pointer, not the data.

add(3, 4)

() triggers the call — CPU jumps to that address and executes. Same as arr[i] actually accessing an element.

Function Pointer — Declaration

Apply the right-left rule. () on the right now means function call instead of array subscript.

3intreturning int
"fp is a pointer to a function taking (int,int) returning int"

1(int,int)right wins: function taking (int,int)
2*left: returning a pointer to
3intint
"fp is a function returning a pointer to int" — very different!
Without (*fp), the (int,int) on the right wins — you get a function returning a pointer, not a pointer to a function.

Assigning and Calling

int add(int a, int b) { return a + b; }
int mul(int a, int b) { return a * b; }

// Declare a function pointer
int (*fp)(int, int);

// Assign — function name (without ()) is its address
fp = add;

// Two equivalent ways to call
fp(3, 4);     // 7  — clean, modern style
(*fp)(3, 4);  // 7  — explicit dereference, same result

// Swap at runtime
fp = mul;
fp(3, 4);     // 12
fp = add and fp = &add are both valid — a function name decays to its address automatically (like array decay), so & is optional, not wrong.
fp(3,4) and (*fp)(3,4) are identical. The compiler handles both. Prefer fp(3,4) — it reads like a normal call.

Callbacks — Passing Behaviour as an Argument

A function pointer can be passed to another function, letting the callee "call back" into your code.

// Apply any integer transformation to every element
void apply(int *arr, int n, int (*transform)(int)) {
    for (int i = 0; i < n; i++) {
        arr[i] = transform(arr[i]);
    }
}

int double_it(int x) { return x * 2; }
int square(int x)    { return x * x; }

int main() {
    int a[] = {1, 2, 3, 4, 5};

    apply(a, 5, double_it);  // {2, 4, 6, 8, 10}
    apply(a, 5, square);     // {4, 16, 36, 64, 100}
}
apply does not know what transform does. You decide at the call site. Same function, completely different behaviour — this is the callback pattern.

qsort — Sorting Without Knowing Your Data

The C standard library has a general-purpose sort built on this exact pattern:

#include <stdlib.h>

void qsort(
    void   *base,                            // pointer to the array
    size_t  nmemb,                           // number of elements
    size_t  size,                            // size of each element
    int   (*compar)(const void*, const void*) // your comparator
);

void* — a pointer with no type. qsort does not know if you are sorting ints, structs, or strings. You tell it two things:

  • How big each element is (size)
  • How to compare two elements (the function pointer)
The comparator must return: negative if a < b, zero if equal, positive if a > b.

qsort — Three Comparators, Same Sort

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

int cmp_asc(const void *a, const void *b) {
    return (*(int*)a - *(int*)b);
}
int cmp_desc(const void *a, const void *b) {
    return (*(int*)b - *(int*)a);
}
int cmp_str(const void *a, const void *b) {
    return strcmp(*(char**)a, *(char**)b);
}

int main() {
    int nums[] = {5, 2, 8, 1, 9};

    qsort(nums, 5, sizeof(int), cmp_asc);
    // → {1, 2, 5, 8, 9}

    qsort(nums, 5, sizeof(int), cmp_desc);
    // → {9, 8, 5, 2, 1}

    char *words[] = {"banana", "apple", "cherry"};
    qsort(words, 3, sizeof(char*), cmp_str);
    // → {"apple", "banana", "cherry"}
}

bsearch — Binary Search, Same Pattern

void *bsearch(
    const void *key,                          // what to search for
    const void *base,                         // the sorted array
    size_t      nmemb,                        // number of elements
    size_t      size,                         // size of each element
    int        (*compar)(const void*, const void*)
);

Returns a pointer to the matching element, or NULL if not found. Array must be sorted first.

int nums[] = {1, 2, 5, 8, 9};  // must be sorted!
int key = 5;

int *result = bsearch(&key, nums, 5, sizeof(int), cmp_asc);

if (result != NULL)
    printf("Found: %d\n", *result);  // Found: 5
else
    printf("Not found\n");
qsort and bsearch use the same comparator — sort first with qsort, then search with bsearch, passing the identical comparison function to both.

What This Pattern Really Is

qsort does not know your data type. You hand it behaviour at runtime via a function pointer.

Without function pointers With function pointers
Write a separate sort for ints, another for strings, another for structs... One sort. Behaviour supplied by the caller.
Changing sort order = rewrite the sort Changing sort order = swap the comparator
Code duplicated everywhere Logic separated from data
This separation of algorithm from comparison policy is called the Strategy Pattern in software design. You will see it everywhere — in every language, in every large codebase.
