A Familiar Frustration

You're solving AtCoder ABC449 - Problem C

Time Limit: 2 seconds

✓ Test Cases 1-5: Accepted

✓ Test Cases 6-10: Accepted

✗ Test Cases 14-21: Time Limit Exceeded (TLE)

Result: 13/21 test cases passed
Your code is correct, but too slow for large inputs!

What is TLE?

Time Limit Exceeded = Your code ran for more than the allowed time

Typical competitive programming limits:

  • 1 second for most problems
  • 2 seconds for harder problems
  • Sometimes 5-10 seconds for very large inputs

But why do small test cases pass while large ones fail?

The Pattern

Test Case Input Size (n) Result Time
1 100 ✓ Pass 0.01s
2 1,000 ✓ Pass 0.08s
3 10,000 ✗ TLE >1s
4-10 100,000+ ✗ TLE >1s

Small inputs work. Large inputs timeout. Your algorithm doesn't scale.

Beyond Competitive Programming

TLE in contests = 1 second limit

But what about real-world problems?

Let's see an extreme example...

The Aadhaar Problem

India has 1 billion (109) Aadhaar numbers

Task: Verify SIM card applicants against this database

Naive Approach: Check every applicant against every Aadhaar number

1 billion applicants × 1 billion database entries

1018 comparisons

How Long Would That Take?

Modern CPU: ~109 operations/second

1018 comparisons ÷ 109 ops/sec

109 seconds

≈ 31 years

By the time it finishes, the data would be obsolete!

The Solution: Binary Search

First, sort the Aadhaar database

Then search by repeatedly halving the space:

109 entries → 500M → 250M → 125M → ...

How many steps to reach 1?

~30 steps

log2(109) ≈ 30

The Shocking Difference

Approach Comparisons Time
Naive (unsorted) 1018 31 years
Binary Search (sorted) 109 × 30 30 seconds

Same data. Different algorithm. 31 years vs 30 seconds.

What is Time Complexity?

The Science of Scalability

Not About Seconds

Wrong Question: "How long does my program take?"
Right Question: "How does runtime grow as input size increases?"

We care about scaling behavior, not absolute time.

Why Not Use Seconds?

  • Different machines run at different speeds
  • Same machine varies (CPU load, temperature, cache)
  • Compiler optimizations differ
  • Programming language matters

Time complexity is hardware-independent

Primitive Operations

To analyze code, we count basic operations:

  • Assign a value: x = 5;
  • Arithmetic: a + b, x * y
  • Compare: if (a < b)
  • Array access: arr[i]
  • Function call/return

Each operation ≈ "1 unit of work"

Example: Linear Search

int linearSearch(int arr[], int n, int target) {
    for (int i = 0; i < n; i++) {      // n iterations
        if (arr[i] == target) {         // 2 ops (access + compare)
            return i;                    // 1 op
        }
    }
    return -1;                          // 1 op
}

Worst case: Target not found, check all n elements

T(n) = setup + loop iterations × operations per iteration

T(n) ≈ 2 + n × 3 = 3n + 2

Big O Notation

The Language of Scalability

The "Munching" Rules

Big O ignores constants and lower-order terms

Example 1: T(n) = 3n + 2

Drop constant (2): 3n

Drop coefficient (3): n

Result: O(n)

More Examples

T(n) = 5n2 + 100n + 50

→ Drop constants and lower terms

→ O(n2)

T(n) = 20n3 + 10n log n + 5

→ Highest order term: n3

→ O(n3)

For large n, only the highest-order term matters!

Why Drop Constants?

Compare: 5n vs 100n

n = 10: 50 vs 1000 (20× difference)

n = 1000: 5000 vs 100,000 (20× difference)

n = 1,000,000: 5M vs 100M (20× difference)

Now compare: n vs n2

n = 10: 10 vs 100 (10× difference)

n = 1000: 1000 vs 1,000,000 (1000× difference)

n = 1,000,000: 1M vs 1,000,000,000,000 (million× difference!)

Growth rate dominates constant factors

Binary Search: How It Works

int binarySearch(int arr[], int n, int target) {
    int left = 0, right = n - 1;
    while (left <= right) {
        int mid = left + (right - left) / 2;
        if (arr[mid] == target) return mid;
        if (arr[mid] < target)
            left = mid + 1;      // Search right half
        else
            right = mid - 1;     // Search left half
    }
    return -1;  // Not found
}

Each iteration halves the search space → O(log n)

O(1): Constant Time

// Array access
int value = arr[100];

// Hash table lookup (average case)
int price = hashTable.get("apple");

// Math operations
int sum = a + b;
int max = (x > y) ? x : y;

Runtime doesn't depend on input size

O(1) vs O(n): A Dramatic Example

Question: What's the fastest way to transfer 3 TB of data?

Internet

Gigabit fiber connection

Upload: limited by ISP

Average: 75 MB/sec

Homing Pigeon

Attach 3× 1TB flash drives

Fly from St. Louis to Nova Scotia

Speed: ~90 MPH

The Results (Jeff Geerling, 2023)

Method Time
Internet (75 MB/sec) 10 hours 54 minutes
Jet-assisted pigeon 6 hours 53 minutes

The pigeon won by 4 hours!

Why Did the Pigeon Win?

Internet = O(n): Time proportional to data size

  • 3 TB takes 11 hours
  • 6 TB would take 22 hours
  • 12 TB would take 44 hours

Pigeon = O(1): Time is constant (up to capacity)

  • 3 TB takes ~7 hours
  • 6 TB takes ~7 hours
  • 38 TB (max capacity) still takes ~7 hours!

Constant time can beat linear time at large scale!

Racing Pigeons: The Numbers

  • Can carry 75 grams
  • MicroSD card: 0.5g, 256GB
  • Capacity: 150 cards = 38.4 TB
  • Speed: 92.5 MPH sustained

Crossover point: ~16 GB
Below 16GB → Internet wins
Above 16GB → Pigeons competitive (up to ~500 miles)

The Key Insight

O(1) doesn't mean "fast" — it means scalable.

The pigeon is "slow" (90 MPH vs speed of light),

but its runtime doesn't grow with data size!

This is the essence of time complexity:
How does runtime scale as input grows?

O(n): Linear Time

// Finding maximum in array
int findMax(int arr[], int n) {
    int max = arr[0];
    for (int i = 1; i < n; i++) {
        if (arr[i] > max) max = arr[i];
    }
    return max;
}

// Printing all elements
for (int i = 0; i < n; i++) {
    printf("%d ", arr[i]);
}

One pass through data → O(n)

O(n2): Quadratic Time

// Bubble Sort
for (int i = 0; i < n-1; i++) {
    for (int j = 0; j < n-i-1; j++) {
        if (arr[j] > arr[j+1]) {
            swap(&arr[j], &arr[j+1]);
        }
    }
}

// Checking for duplicates (naive)
for (int i = 0; i < n; i++) {
    for (int j = i+1; j < n; j++) {
        if (arr[i] == arr[j]) return true;
    }
}

Nested loop → O(n2)

Analysis Techniques

Mathematical Tools

Summation: 1 + 2 + 3 + ... + n

This pattern appears in many algorithms:

// Selection Sort inner loop
for (int i = 0; i < n; i++) {
    for (int j = i+1; j < n; j++) {  // n-1, n-2, ..., 1 iterations
        // ...
    }
}

Total iterations: (n-1) + (n-2) + ... + 1

Formula: 1 + 2 + ... + n = n(n+1)/2

= n2/2 + n/2

→ O(n2)

Nested Loops

Rule: Multiply the complexities

// Outer loop: O(n)
for (int i = 0; i < n; i++) {
    // Inner loop: O(n)
    for (int j = 0; j < n; j++) {
        printf("%d,%d ", i, j);
    }
}

Total: O(n) × O(n) = O(n2)

Divide and Conquer

If problem size is cut in half each step → O(log n)

n = 1,000,000,000 (1 billion)

→ 500,000,000

→ 250,000,000

→ 125,000,000

→ ...

→ 1

Steps: log2(109) ≈ 30

Why log n is Amazing

n log2 n
1,000 ~10
1,000,000 ~20
1,000,000,000 ~30
1,000,000,000,000 ~40

Logarithmic growth is incredibly slow!

Why It Matters

The Real-World Impact

n = 100:

Algorithm Operations
O(n) 100

Real Companies, Real Impact

  • Google: Processes billions of searches daily
    → O(n2) algorithm would be unusable
  • Facebook: 3 billion users
    → Friend suggestions must be O(n log n) or better
  • Netflix: Recommending from millions of titles
    → Must respond in <1 second
  • Stock Exchanges: Microsecond-level trading
    → Even O(n log n) can be too slow

Algorithm choice determines what's possible at scale

The Sorting Benchmark

Sorting is so common that it defines the "efficiency baseline"

Algorithm Complexity Verdict
Bubble Sort, Selection Sort O(n2) Poor
Merge Sort, Quick Sort, Heap Sort O(n log n) Efficient

O(n log n) is considered the "gold standard" for comparison-based sorting

Key Takeaways

  • Time complexity measures scalability, not seconds
  • Big O notation ignores constants and lower-order terms
  • Growth rate dominates for large inputs
  • O(1) < O(log n) < O(n) < O(n log n) < O(n2) < O(2n)
  • Binary search (O(log n)) beats linear search (O(n))
  • Efficient sorting (O(n log n)) beats naive sorting (O(n2))
  • Algorithm choice can mean 31 years vs 30 seconds!
