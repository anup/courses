Linked Lists

Part II — Pointer-Based Linked Lists

ES 248 · Introduction to DSA in C

Part 1

From Arrays to Pointers

Last Time: Two Separate Arrays

We stored a linked list using data[] and nxt[] — two parallel arrays.

Each "node" was split across two memory locations.

What if we bundled a node's data
and its "next" into a single object?

That's what a struct does.

And the integer index becomes a pointer to that struct.

The Node — A Self-Referential Struct

struct Node {
    int data;
    struct Node *next;  // pointer to the next node
};

Array-based (Part I)

data[i]  → the value

nxt[i]   → integer index

head    → int

Pointer-based (Part II)

node->data  → the value

node->next  → pointer to struct

head       → struct Node *

Same logic. Same operations. But now each node is one self-contained block in memory, allocated with malloc.

Memory Layout

Each node lives at its own malloc'd address. Connected only by pointers.

head = 0x7A20
[0x7A20] data=8 | next=0x3F10 → [0x3F10] data=15 | next=0x5C48 → [0x5C48] data=23 | next=NULL
Addresses are not consecutive — each malloc returns whatever is available. The pointers stitch them together.
Compare: in Part I, everything lived inside data[0..9] and nxt[0..9] — fixed, pre-allocated arrays.

Part 2

Implement Together

1 — createNode(int value)

Task: Allocate a new node, set its data, set next to NULL. 
struct Node *createNode(int value) {
    struct Node *node = malloc(sizeof(struct Node));
    if (node == NULL) {
        printf("Error: malloc failed\n");
        return NULL;
    }
    node->data = value;
    node->next = NULL;
    return node;
}
This is the pointer-based equivalent of fetch_from_avail() from Part I. Instead of grabbing a slot from our avail chain, we ask the OS for memory.

2 — insertFront(struct Node **head, int val)

Task: Create a node and prepend it to the list.

Why struct Node **head? We need to modify the caller's head pointer. A single pointer (struct Node *head) would only change the local copy.

void insertFront(struct Node **head, int val) {
    struct Node *node = createNode(val);
    if (node == NULL) return;

    node->next = *head;   // new node points to old first node
    *head = node;          // head now points to new node
}
Compare Part I: nxt[slot] = head; head = slot; — same two assignments, different syntax.

3 — insertEnd(struct Node **head, int val)

Task: Traverse to the last node, then link the new node after it. 
void insertEnd(struct Node **head, int val) {
    struct Node *node = createNode(val);
    if (node == NULL) return;

    if (*head == NULL) {       // empty list
        *head = node;
        return;
    }

    struct Node *curr = *head;
    while (curr->next != NULL)
        curr = curr->next;     // walk to the last node

    curr->next = node;         // link new node at the end
}

4 — deleteNode(struct Node **head, int val)

Task: Find the node, unlink it, free its memory. 
void deleteNode(struct Node **head, int val) {
    struct Node *prev = NULL, *curr = *head;

    while (curr != NULL && curr->data != val) {
        prev = curr;
        curr = curr->next;
    }
    if (curr == NULL) return;          // not found

    if (prev == NULL) *head = curr->next;   // removing head
    else              prev->next = curr->next;  // bypass

    free(curr);    // return memory to the OS
}
free(curr) is the pointer-based equivalent of return_to_avail(). The OS reclaims the memory.

5 — traverse(struct Node *head)

Task: Walk the list, print each value. 
void traverse(struct Node *head) {
    struct Node *curr = head;
    while (curr != NULL) {
        printf("%d ", curr->data);
        curr = curr->next;
    }
    printf("\n");
}
Notice: traverse takes Node * (single pointer) — it only reads, never modifies head.

6 — freeList(struct Node **head)

Task: Free every node. Set head to NULL. Without this, you have a memory leak. 
void freeList(struct Node **head) {
    struct Node *curr = *head;
    while (curr != NULL) {
        struct Node *next = curr->next;  // save next BEFORE freeing
        free(curr);
        curr = next;
    }
    *head = NULL;
}
Why save curr->next first? After free(curr), accessing curr->next is undefined behavior — the memory may already be reused.

Part 3

Why Linked Lists Still Matter

The Honest Truth

Arrays are usually faster on modern hardware.

CPUs love sequential memory — cache lines, prefetching, branch prediction all favor arrays.

So why learn linked lists?

Because the concept of linking nodes with pointers is everywhere.

Data StructureWhat It Really Is
TreesA node with two next pointers (left, right)
Hash tablesArray of buckets, each bucket is a linked list (chaining)
GraphsAdjacency list = one linked list per vertex
Memory allocatorsFree list — you already built one in Part I (avail)
Master the linked list, and trees, graphs, and hash tables will feel like extensions — not new topics.

Part 4

Some Interesting Problems

Problem 1 — Detecting a Cycle

Someone corrupted your list. The last node points back to a middle node instead of NULL.

1 → 2 → 3 → 4 → 5 → 3 → 4 → 5 → 3 → ...

How do you detect this — without extra memory?

One approach: store every visited node's address in an array. Before visiting a node, check if you've seen it before.

That works — but uses O(n) extra memory.

Floyd's Tortoise and Hare: Two pointers, both starting at head.
slow moves 1 step. fast moves 2 steps.
If there's a cycle, fast gains 1 step per iteration — they must meet.
If no cycle, fast hits NULL.

Floyd's Cycle Detection — Code

int hasCycle(struct Node *head) {
    struct Node *slow = head;
    struct Node *fast = head;

    while (fast != NULL && fast->next != NULL) {
        slow = slow->next;          // 1 step
        fast = fast->next->next;    // 2 steps

        if (slow == fast)
            return 1;    // they met — cycle exists
    }
    return 0;    // fast reached NULL — no cycle
}
O(n) time, O(1) space. No extra array, no marking nodes, no extra memory. Just two pointers.
Why must they meet? In the cycle, fast closes the gap by 1 each iteration. After at most one full loop, they collide.

Problem 2 — Reverse a Linked List

The most asked linked list interview question.

Before:  1 → 2 → 3 → 4 → NULL

After:   4 → 3 → 2 → 1 → NULL

Idea: Walk through the list. At each node, flip the next pointer backward.

Three pointers: prev, curr, next.

Step 1:  NULL ← 1    2 → 3 → 4 → NULL

Step 2:  NULL ← 1 ← 2    3 → 4 → NULL

Step 3:  NULL ← 1 ← 2 ← 3    4 → NULL

Step 4:  NULL ← 1 ← 2 ← 3 ← 4

Reverse — Code

void reverse(struct Node **head) {
    struct Node *prev = NULL;
    struct Node *curr = *head;

    while (curr != NULL) {
        struct Node *next = curr->next;   // save next
        curr->next = prev;                // flip the arrow
        prev = curr;                      // advance prev
        curr = next;                      // advance curr
    }
    *head = prev;    // prev is now the first node
}
Four lines in the loop. O(n) time, O(1) space. No extra list, no recursion needed.

Reverse — Can We Do It Recursively?

The iterative version is clean. Why bother with recursion?

Why this matters: Reversing a linked list is the canonical problem where you can see the exact same logic written in two completely different recursive styles. Understanding the difference teaches you how to think recursively.

The recursive question: Break it into one step + the rest.

1 → 2 → 3 → 4 → NULL

"If I trust recursion to reverse 2→3→4, what do I need to do with 1?"

→ After recursion:  4 → 3 → 2 → (1 is dangling, still pointing at 2)

→ Fix: make 2 point back at 1, then cut 1's forward link.

Two ways to arrange this work: fix on the way back, or carry state going forward.

Approach 1 — Fix on the Way Back

struct Node* reverse_rec(struct Node *head) {
    // Base case: empty list or last node — return as the new head
    if (head == NULL || head->next == NULL) return head;

    // Trust recursion to reverse everything after head
    struct Node *new_head = reverse_rec(head->next);

    // NOW on the way back: head->next still points at the old second node
    // That node should now point BACK at head
    head->next->next = head;
    head->next = NULL;    // head is now the tail — cut its forward link

    return new_head;      // bubble up the true new head all the way
}

reverse_rec(1) calls reverse_rec(2) calls reverse_rec(3) calls reverse_rec(4)

  ← returns 4 as new_head.  Now fix: 4→3 already done, fix 3→2: 3.next.next = 3 → 3→2→3? No — 2.next = 3

  ← Each frame fixes one arrow, then passes new_head up unchanged.

The work happens on the return trip. You must hold the whole call stack in memory before doing anything. Call stack depth: O(n).

Approach 2 — Carry State Forward (Tail Recursive)

struct Node* reverse_tail(struct Node *curr, struct Node *prev) {
    if (curr == NULL) return prev;   // prev is now the new head — done

    struct Node *next = curr->next;  // save next before we overwrite it
    curr->next = prev;               // flip the arrow right now

    return reverse_tail(next, curr); // move forward, carrying work done so far
}

// Caller: head = reverse_tail(head, NULL);

reverse_tail(1, NULL) → flips 1, calls reverse_tail(2, 1)

reverse_tail(2, 1)  → flips 2, calls reverse_tail(3, 2)

reverse_tail(3, 2)  → flips 3, calls reverse_tail(4, 3)

reverse_tail(4, 3)  → flips 4, calls reverse_tail(NULL, 4) → returns 4

The work happens going forward. The recursive call is the last thing each frame does — that is what tail recursion means. The result just passes straight back up without further processing.

Two Styles — One Problem

Classic RecursiveTail Recursive
Work happensOn the return trip (back up)Going forward (down)
State carriedImplicitly in the call stackExplicitly as a parameter (prev)
Reads likePure recursion — trust, then fixIterative loop in disguise
Stack depthO(n)O(n) — same in C*
Base case returnsThe new head (bubbles up)The new head (passes straight through)
*Some languages/compilers optimize tail calls to O(1) stack space. C does not guarantee this.
Which to learn first? Classic — it forces you to think about what the call stack is doing. Once that clicks, tail recursion reveals itself as "I moved the state from the stack into a parameter."
Pattern you will see again: Any time a recursive function carries an accumulator forward (running total, running max, built-up result), that is tail recursion in spirit.

Problem 3 — The Josephus Problem

Story (1st century AD): N soldiers stand in a circle. Starting from a chosen position, every k-th person is eliminated. The last one standing survives.

Example: N=6, k=3

Circle: 1, 2, 3, 4, 5, 6  (standing in a circle)
Eliminate 3:  1, 2, _, 4, 5, 6
Eliminate 6:  1, 2, _, 4, 5, _
Eliminate 4:  1, 2, _, _, 5, _
Eliminate 2:  1, _, _, _, 5, _
Eliminate 5:  1 survives

Your task: Write a function that takes N and k, and returns the survivor's number.
Fun fact: Flavius Josephus (37 AD) reportedly used this reasoning to choose the safe position and survive.

Josephus — Array Solution

The natural first approach: use an array and mark eliminated people.

int josephus_array(int n, int k) {
    int alive[n];
    for (int i = 0; i < n; i++)
        alive[i] = 1;          // everyone starts alive

    int remaining = n;
    int pos = 0;                // current position in the circle

    while (remaining > 1) {
        // Count k alive people
        int count = 0;
        while (count < k) {
            if (alive[pos])
                count++;
            if (count < k)
                pos = (pos + 1) % n;   // wrap around
        }
        printf("Eliminated %d\n", pos + 1);
        alive[pos] = 0;
        remaining--;
        pos = (pos + 1) % n;
    }

    // Find the survivor
    for (int i = 0; i < n; i++)
        if (alive[i]) return i + 1;
    return -1;
}

Josephus — What's Wrong With the Array?

  • Skipping dead people: Every time you count k, you must step over eliminated slots. As more people die, you skip more and more empty slots.
  • Wrap-around logic: (pos + 1) % n everywhere — easy to get wrong.
  • Wasteful: The array never shrinks. With 1000 people and 999 eliminated, you're still walking past 999 dead slots to find the last two alive.
What if we could actually remove a person from the circle when they're eliminated — so we never skip anyone?

Josephus — A Circular Linked List

Make the last node's next point back to the first node. A circle — no NULL, no end.

1 → 2 → 3 → 4 → 5 → 6 → (back to 1)

  • To eliminate someone: prev->next = curr->next — they're gone from the circle. O(1).
  • No skipping dead slots. Every step lands on a living person.
  • Counting k steps means exactly k pointer follows.
  • One person left? curr->next == curr — done.
The circle shrinks as people are eliminated. No wasted work.

Josephus — Linked List Code

int josephus(int n, int k) {
    // Build circular linked list: 1 → 2 → ... → n → (back to 1)
    struct Node *head = createNode(1);
    struct Node *curr = head;
    for (int i = 2; i <= n; i++) {
        curr->next = createNode(i);
        curr = curr->next;
    }
    curr->next = head;   // make it circular

    // Eliminate every k-th person
    struct Node *prev = curr;   // prev starts at the last node
    curr = head;

    while (curr->next != curr) {   // more than one person left
        // Count k-1 steps (curr lands on the k-th person)
        for (int i = 1; i < k; i++) {
            prev = curr;
            curr = curr->next;
        }
        // Eliminate curr
        prev->next = curr->next;
        printf("Eliminated %d\n", curr->data);
        struct Node *temp = curr;
        curr = curr->next;
        free(temp);
    }

    int survivor = curr->data;
    free(curr);
    return survivor;
}

Josephus — Array vs. Linked List

ArrayCircular Linked List
Eliminate someoneMark as dead, still takes spaceRemove from circle, free memory
Count k stepsMust skip dead slotsEvery step lands on a living person
Wrap around(pos+1) % nAutomatic — it's a circle
Detect last personScan array for the one alivecurr->next == curr
This pattern — processes arranged in a circle, each getting a turn — is exactly round-robin scheduling in operating systems.

Josephus — Time Complexity

We eliminate n−1 people. k is the counting number — every k-th person is eliminated. How much work does each elimination take?

Linked list: dead nodes are physically removed. Every step lands on a living person. Each elimination costs exactly k steps.   (n−1) eliminations × k steps = O(n × k).
Array: dead slots stay in the array, just marked. To count k alive people you must skip over dead ones. Near the end — say 2 people left in an array of n — a single elimination might walk past ~n dead slots. Worst case: O(n²).
Elimination #Array steps (skipping dead)Linked list steps
1st  (all n alive)kk
Midway  (n/2 dead)~2k  (skip 1 dead per alive)k
Near end  (2 alive)up to ~n  (walk past n−2 dead)k

Example: n = 100,000 and k = 2

SolutionOperations
Array (worst case)O(n²) = 10,000,000,000
Circular linked listO(n × k) = 200,000
The linked list isn't just cleaner code — it is genuinely faster, because it erases dead work instead of carrying it forward.

Recap

  • Struct + pointer = one self-contained node, dynamically allocated
  • Same operations as the array version — just different syntax
  • Linked list = the building block for trees, graphs, hash tables
